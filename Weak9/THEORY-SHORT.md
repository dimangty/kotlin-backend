# Неделя 9 — краткая теория

**Тема:** аутентификация, авторизация и безопасность API.
**Результат:** безопасный доступ к данным без смешивания security и бизнес-логики.

---

## 1. Пароли

- Хранить только **хеш**, никогда пароль и никогда обратимое шифрование.
- Алгоритмы: **Argon2id** (предпочтительно), **bcrypt** (проверенный, но лимит 72 байта), scrypt. Никогда не `MD5`/`SHA-256` сами по себе — они слишком быстрые.
- **Salt** уникален для каждого пароля и хранится вместе с хешем (в самой строке хеша). Защищает от rainbow tables.
- **Параметры стоимости** (bcrypt `cost`, Argon2 `memory`/`iterations`/`parallelism`) подбираются так, чтобы проверка занимала ~100-250 мс на вашем железе.
- `DelegatingPasswordEncoder` в Spring Security хранит префикс алгоритма (`{bcrypt}...`) → миграция на новый алгоритм без сброса паролей.

## 2. Аутентификация и токены

| | Access token | Refresh token |
|---|---|---|
| Время жизни | минуты (5-15) | дни/недели |
| Где используется | каждый запрос | только `/auth/refresh` |
| Отзыв | практически невозможен | обязателен |
| Хранение на сервере | не хранится (stateless) | хранится (id, хеш, статус) |

**Rotation:** каждый `refresh` выдаёт новую пару и инвалидирует старый refresh. Повторное использование уже использованного refresh — сигнал кражи → отозвать всю цепочку сессий пользователя.

**Что нельзя класть в JWT payload:** пароли, персональные данные, номера счетов, права, которые могут измениться. JWT читаем любым, у кого он есть — он только подписан, не зашифрован.

Проверять обязательно: подпись, `exp`, `iss`, `aud`, алгоритм (запретить `none` и подмену `RS256`→`HS256`).

## 3. Spring Security

```
FilterChainProxy
 └ SecurityFilterChain
     ├ CorsFilter
     ├ CsrfFilter                (для stateless API с Bearer — отключается)
     ├ ваш JwtAuthenticationFilter → SecurityContextHolder
     ├ AuthorizationFilter       → решение о доступе
     └ ExceptionTranslationFilter → 401 / 403
```

- `SecurityContextHolder` — ThreadLocal; при переходе в другой поток контекст теряется (важно на неделе 11).
- `401` — не аутентифицирован; `403` — аутентифицирован, но нет прав.

## 4. RBAC и ownership

- **RBAC** отвечает на «какая у тебя роль»: `hasRole('ADMIN')`.
- **Ownership** отвечает на «твой ли это объект»: `account.userId == principal.id`.

Роли **не заменяют** проверку владения. Отсутствие ownership-проверки — это BOLA (Broken Object Level Authorization), риск №1 в OWASP API Security Top 10.

Правильное место проверки — сервис или сам запрос:

```sql
SELECT ... FROM accounts WHERE id = :id AND user_id = :currentUser
```

Так невозможно забыть проверку, и не возникает утечки «объект существует».

## 5. OWASP API Top 10 — минимум

| Риск | Проявление | Защита |
|---|---|---|
| BOLA | доступ к чужому `id` | ownership-проверка в запросе |
| Broken authentication | слабые токены, нет отзыва | rotation, короткий access |
| Excessive data exposure | entity наружу целиком | DTO с явными полями |
| Injection | конкатенация SQL | параметризованные запросы |
| Rate abuse | brute force логина | rate limit, задержки, блокировка |
| Security misconfiguration | stack trace в ответе | единый error contract |

## 6. Секреты и audit

- Секреты — только из переменных окружения/секрет-хранилища, никогда в репозитории.
- Ключи подписи ротируются: `kid` в заголовке JWT позволяет держать несколько активных ключей.
- **Audit trail** — неизменяемая история: кто (actor), что, над чем, когда, `requestId`. Только `INSERT`.
- В логи не попадают: пароли, токены, номера карт, полные тела запросов с ПДн.

## Контрольные вопросы

1. **Зачем salt, если хеш и так необратим?** Без соли одинаковые пароли дают одинаковые хеши, и предвычисленные таблицы вскрывают их массово.
2. **Почему access token короткоживущий?** Его нельзя отозвать: единственная защита от украденного токена — быстрое истечение.
3. **Что произойдёт при повторном использовании refresh token?** Это признак кражи: система должна инвалидировать всю цепочку и потребовать повторный вход.
4. **Чем `401` отличается от `403`?** `401` — личность не установлена; `403` — установлена, но прав недостаточно.
5. **Почему `404` иногда предпочтительнее `403`?** Чтобы не подтверждать существование чужого объекта.
