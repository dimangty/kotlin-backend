# Неделя 9 — подробная теория

Аутентификация, авторизация и безопасность API.

> Security — это слой, который должен быть **снаружи** бизнес-логики и при этом проверяться теми же интеграционными тестами. Главная проверка недели — негативные тесты: чужой пользователь не может получить ваш счёт, а отозванный токен перестаёт работать немедленно.

---

## 1. Хранение паролей

### 1.1 Требования

Пароль хранится только как **хеш медленной функции с солью**. Не шифрование (обратимо), не быстрый хеш (`SHA-256` перебирается на GPU миллиардами в секунду).

### 1.2 Алгоритмы

| Алгоритм | Параметры | Замечания |
|---|---|---|
| **Argon2id** | `memory`, `iterations`, `parallelism` | современный выбор; memory-hard, устойчив к GPU/ASIC |
| **bcrypt** | `cost` (log2 итераций) | проверенный временем; **обрезает вход до 72 байт** |
| **scrypt** | `N`, `r`, `p` | memory-hard, реже используется |
| PBKDF2 | итерации | приемлем, где требуется FIPS |

Ориентир по параметрам: проверка пароля должна занимать **100-250 мс** на вашем железе. Меньше — слишком дёшево для атакующего; больше — вы сами создали себе вектор DoS (каждый запрос логина съедает CPU).

### 1.3 Salt и pepper

- **Salt** — случайное значение на каждый пароль, хранится рядом с хешем (в строке формата `$argon2id$v=19$m=...,t=...,p=...$<salt>$<hash>`). Делает бесполезными rainbow tables и скрывает совпадение паролей у разных пользователей.
- **Pepper** — секрет приложения, добавляемый ко всем паролям, хранится **вне базы** (переменная окружения/HSM). Защищает при утечке одной только базы. Опционально, но полезно; усложняет ротацию.

### 1.4 Миграция алгоритма

```kotlin
@Bean
fun passwordEncoder(): PasswordEncoder = PasswordEncoderFactories.createDelegatingPasswordEncoder()
// хранит "{bcrypt}$2a$10$..." или "{argon2}$argon2id$..."
```

Префикс позволяет держать в базе несколько форматов одновременно. При успешном логине, если хеш в старом формате, перехешируйте пароль в новый — миграция произойдёт постепенно, без сброса паролей.

### 1.5 Защита процесса логина

- **Timing:** отвечать одинаково на «пользователь не найден» и «неверный пароль», и тратить сопоставимое время (выполнить фиктивную проверку хеша). Иначе endpoint становится оракулом существования аккаунтов.
- **Rate limiting** по IP и по логину: экспоненциальная задержка, временная блокировка, CAPTCHA при превышении.
- Не сообщать в ответе, какой именно фактор неверен.

## 2. Токены и сессии

### 2.1 Модель access/refresh

| | Access | Refresh |
|---|---|---|
| Живёт | 5-15 минут | дни-недели |
| Передаётся | в каждом запросе (`Authorization: Bearer`) | только на `/auth/refresh` |
| Хранится на сервере | нет | **да** (id, хеш токена, user_id, статус, `expires_at`, device/ip) |
| Отзыв | практически невозможен | обязателен |

Смысл разделения: access делает авторизацию дешёвой (без похода в базу), refresh даёт контроль (отзыв, выход со всех устройств).

### 2.2 JWT: что проверять

JWT **подписан, но не зашифрован** — payload читает любой, кто его получил.

Обязательные проверки на сервере:

1. **Подпись** и **алгоритм из белого списка**. Классические атаки: `alg: none` и подмена `RS256` → `HS256` (публичный ключ используется как HMAC-секрет). Алгоритм фиксируется в коде, а не берётся из токена.
2. `exp` — истечение; `nbf` — не раньше; допуск на рассинхронизацию часов — секунды, не минуты.
3. `iss` и `aud` — токен выпущен вами и предназначен вам.
4. `jti` — идентификатор, если ведёте denylist.
5. `kid` — какой ключ использовать (нужен для ротации ключей).

Чего **не** должно быть в payload: пароли и их хеши, персональные данные, номера счетов и карт, права, способные измениться до истечения токена.

### 2.3 Refresh rotation и обнаружение кражи

Протокол:

1. `/auth/refresh` принимает refresh token.
2. Сервер находит запись, проверяет статус и срок.
3. Выдаёт **новую пару** и помечает старый refresh как `USED`, связывая с новым (цепочка).
4. Если приходит refresh, уже помеченный `USED` — это **признак кражи**: кто-то использует копию. Реакция: инвалидировать всю цепочку (все потомки), потребовать повторный вход, записать событие в audit.

Хранить в базе стоит **хеш** refresh-токена, а не сам токен: утечка базы не должна давать готовые к использованию токены.

### 2.4 Выход

- `logout` — пометить refresh как revoked.
- «Выйти со всех устройств» — отозвать все refresh пользователя.
- Access token останется валидным до `exp` — это цена stateless-подхода. Если требуется мгновенный отзыв, нужен denylist по `jti` (обычно в Redis с TTL до `exp`) или короткий `exp`.

### 2.5 Хранение на клиенте

Для мобильного клиента — Keychain/Keystore. Для браузера: `httpOnly`, `Secure`, `SameSite` cookie предпочтительнее `localStorage` (недоступно из JS → нет XSS-кражи), но тогда возвращается необходимость CSRF-защиты.

## 3. Spring Security

### 3.1 Filter chain

```
FilterChainProxy
 └ SecurityFilterChain (по matcher'у)
     ├ CorsFilter
     ├ CsrfFilter
     ├ JwtAuthenticationFilter (ваш)  → SecurityContextHolder.setAuthentication(...)
     ├ AuthorizationFilter            → решение по правилам
     └ ExceptionTranslationFilter     → AuthenticationEntryPoint (401) / AccessDeniedHandler (403)
```

Всё, что происходит здесь, — **до** контроллера. Бизнес-код не должен знать про фильтры.

### 3.2 Конфигурация для stateless API

```kotlin
@Bean
fun filterChain(http: HttpSecurity, jwtFilter: JwtAuthenticationFilter): SecurityFilterChain =
    http
        .csrf { it.disable() }                               // stateless + Bearer → CSRF неприменим
        .sessionManagement { it.sessionCreationPolicy(STATELESS) }
        .authorizeHttpRequests {
            it.requestMatchers("/auth/**", "/health").permitAll()
              .requestMatchers("/actuator/**").hasRole("OPS")
              .anyRequest().authenticated()                  // deny by default
        }
        .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter::class.java)
        .exceptionHandling { ... }                           // единый формат ошибок
        .build()
```

Ключевое — **`anyRequest().authenticated()`**: политика по умолчанию должна быть «запрещено», иначе новый endpoint окажется открытым по недосмотру.

Отключение CSRF законно **только** для stateless API с токеном в заголовке. Если авторизация идёт через cookie, CSRF-защита обязательна.

### 3.3 Principal

```kotlin
data class AppPrincipal(val userId: Long, val roles: Set<String>)

val principal = SecurityContextHolder.getContext().authentication.principal as AppPrincipal
```

`SecurityContextHolder` — ThreadLocal. При переходе в другой поток (`@Async`, корутины, `CompletableFuture`) контекст **теряется**. Это прямо касается недели 11: передавайте `userId` явным параметром, а не полагайтесь на ThreadLocal.

## 4. Авторизация: RBAC и ownership

### 4.1 Два разных вопроса

- **RBAC**: «какая у тебя роль?» → `hasRole('ADMIN')`, `hasAuthority('payments:write')`.
- **Ownership**: «этот объект твой?» → `account.userId == principal.userId`.

Роль `USER` не означает право на **этот** счёт. Пропуск ownership-проверки — самая частая и самая дорогая уязвимость API (BOLA).

### 4.2 Куда помещать проверку

**Плохо** — проверка после загрузки, разбросанная по контроллерам:

```kotlin
val account = repo.findById(id)
if (account.userId != principal.userId) throw AccessDenied()   // легко забыть в новом endpoint
```

**Лучше** — ownership встроен в запрос:

```kotlin
@Query("select a from Account a where a.id = :id and a.userId = :userId")
fun findOwned(id: Long, userId: Long): Account?
```

или в SQL:

```sql
SELECT ... FROM accounts WHERE id = :id AND user_id = :currentUser;
```

Забыть невозможно, и заодно не возникает утечки «объект существует, но чужой». Для сложных случаев — `@PreAuthorize("@accessChecker.ownsAccount(#id, principal)")`, но помните: method security тоже работает через прокси, со всеми ограничениями self-invocation.

### 4.3 403 или 404

- `403` честнее по семантике, но подтверждает существование объекта.
- `404` не раскрывает ничего, но затрудняет отладку.

Для финтеха обычно выбирают `404` на чужие объекты. Главное — быть последовательным и задокументировать выбор.

## 5. OWASP API Security: практический минимум

| Риск | Как выглядит у нас | Защита |
|---|---|---|
| **BOLA** | `GET /accounts/{id}` без проверки владельца | ownership в запросе + негативный тест |
| **Broken authentication** | долгий access, нет отзыва refresh | короткий access, rotation, denylist |
| **Broken object property authorization** | `PATCH` позволяет менять `role`/`balance` | явный DTO с белым списком полей |
| **Unrestricted resource consumption** | `limit=1000000`, brute force логина | ограничение размера страницы, rate limit, таймауты |
| **Broken function level authorization** | админский endpoint без роли | deny by default + тест на каждую роль |
| **Excessive data exposure** | entity наружу с `passwordHash` | DTO с явными полями |
| **Injection** | конкатенация SQL | только параметризованные запросы |
| **Security misconfiguration** | stack trace, открытый Actuator | единый error contract, Actuator под ролью |

Mass assignment заслуживает отдельного упоминания: DTO запроса должен содержать **только** те поля, которые клиент вправе менять. Приём entity/полного объекта из тела запроса — прямой путь к повышению привилегий.

## 6. Секреты, ключи, audit

### 6.1 Секреты

- Только переменные окружения или секрет-хранилище. Никогда в `application.yaml` в репозитории.
- Отдельные секреты для каждого окружения.
- Утёкший секрет считается скомпрометированным навсегда — ротация, а не «мы удалили коммит».
- Полезно: сканер секретов в CI (неделя 13).

### 6.2 Ротация ключей подписи

`kid` в заголовке JWT позволяет держать несколько активных ключей. Процедура: добавить новый ключ (проверка принимает оба) → начать подписывать новым → дождаться истечения токенов, подписанных старым → удалить старый. Это тот же expand-contract, что и в миграциях.

### 6.3 Audit trail

```sql
CREATE TABLE audit_events (
    id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    actor_id    bigint,
    action      text NOT NULL,           -- LOGIN, TRANSFER_CREATED, TOKEN_REVOKED
    object_type text,
    object_id   text,
    request_id  text,
    ip          inet,
    created_at  timestamptz NOT NULL DEFAULT now(),
    payload     jsonb
);
```

Свойства: **только `INSERT`** (как ledger), содержит `requestId` для связи с логами (неделя 12), пишется в той же транзакции, что и действие. Не содержит секретов и полных персональных данных.

Что писать обязательно: вход/выход, неудачные попытки входа, выдача и отзыв токенов, изменение прав, все финансовые операции.

### 6.4 Логи

Не логировать: пароли, токены (даже частично — префикс тоже помогает атакующему), номера карт, полные тела запросов с ПДн, содержимое `Authorization`.

Практический приём — белый список полей для логирования и маскирование по умолчанию. На неделе 12 это станет частью MDC-контракта.

## 7. Транспорт и заголовки

- **HTTPS обязателен**; HSTS.
- **CORS** настраивается явным списком origin; `allowedOrigins("*")` вместе с credentials недопустимо.
- Заголовки: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy`, `Content-Security-Policy` (для страниц).
- Не отдавать версию сервера и фреймворка.

## 8. Лаборатория недели

1. Регистрация, login, refresh с rotation, logout, «выйти со всех устройств».
2. Хранение пароля через `DelegatingPasswordEncoder`; замерить время проверки и подобрать параметры.
3. Защитить операции над счётом ownership-проверкой **внутри запроса**.
4. Негативные интеграционные тесты (обязательная часть):
   - чужой пользователь получает `404`/`403` на чужой счёт;
   - запрос без токена → `401`;
   - истёкший токен → `401`;
   - токен с изменённой подписью → `401`;
   - `alg: none` → `401`;
   - отозванный refresh → `401`;
   - повторное использование refresh → инвалидация цепочки;
   - обычный пользователь на админском endpoint → `403`.
5. Добавить audit event на все критичные действия, связать с `requestId`.
6. Проверить, что в логах нет секретов: намеренно поискать токен в выводе.
7. Rate limit на `/auth/login` и проверка, что перебор замедляется.

## 9. Типичные ошибки недели

1. Проверка роли вместо проверки владения.
2. Долгоживущий access token без возможности отзыва.
3. Refresh token в базе в открытом виде.
4. Персональные данные и права в payload JWT.
5. Алгоритм подписи берётся из самого токена.
6. `permitAll()` по умолчанию, вместо `authenticated()`.
7. Приём entity в теле запроса → mass assignment.
8. Разные ответы на «нет пользователя» и «неверный пароль».
9. Логирование `Authorization` целиком.
10. Security-проверки внутри бизнес-логики вперемешку с правилами домена.
11. Отсутствие негативных тестов — «позитивный тест проходит, значит безопасно».

---

## 10. Критерий готовности

- Интеграционные тесты подтверждают запрет доступа к чужим ресурсам.
- Refresh token можно отозвать без ожидания истечения access token.
- Можете объяснить, что нельзя помещать в JWT payload и почему.
- Security не перемешана с бизнес-логикой: доступ решается в фильтре и в запросе, а не в середине сервиса.
- Audit фиксирует все критичные действия и связывается с `requestId`.
