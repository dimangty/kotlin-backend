# Неделя 11. Корутины, внешние сервисы и отказоустойчивость

**Результат недели:** корректно работать с таймаутами и отменой, не удерживая транзакцию базы во время сетевого вызова.

Неделя строится вокруг одного неприятного факта: **timeout не означает, что операция не выполнилась.** Клиент не получил ответ - и всё, что он знает. Деньги при этом могли уже уйти. Весь код недели - следствие этого факта.

## Три состояния вместо двух

| Состояние | Что известно | Что делать |
|---|---|---|
| `COMPLETED` | внешняя операция подтверждена | вернуть сохранённый результат |
| `FAILED` | внешняя система явно отказала | можно повторять как новую попытку |
| `RESERVED` | результат неизвестен: timeout, отмена, обрыв | **не считать отказом**, разбирать сверкой (reconciliation) |

Ошибка, которую ловит эта модель: пометить неизвестный результат как `FAILED`, отдать клиенту ошибку, он повторит - и заплатит дважды. Поэтому в [PaymentCoordinator.kt](src/main/kotlin/study/week11/PaymentCoordinator.kt) `CancellationException` перебрасывается, не трогая состояние, а `FAILED` ставится только для явных ошибок.

## Теория и где она в проекте

| Тема | Где смотреть |
|---|---|
| Транзакция и сеть не пересекаются | `PaymentCoordinator.pay` - два коротких сохранения вокруг сетевого вызова, а не одна длинная транзакция поверх него |
| Timeout | `withTimeout(500)` только вокруг вызова gateway |
| Structured concurrency и cancellation | `CancellationException` перехватывается отдельно и перебрасывается: отмена не поглощается |
| Идемпотентность через границу систем | один и тот же key уходит и в repository, и в gateway ([DemoGateway.kt](src/main/kotlin/study/week11/DemoGateway.kt)) |
| Защита от параллельного повтора | `Mutex` на ключ операции; комментарий в коде честно говорит, что для нескольких instance эту роль обязаны выполнять UNIQUE constraint и state machine в базе |
| Совпадение payload | тот же key с другой суммой отклоняется, а не молча возвращает старый результат |
| Тесты | [PaymentCoordinatorTest.kt](src/test/kotlin/study/week11/PaymentCoordinatorTest.kt) - повтор, параллельный повтор, конфликт payload, отмена оставляет `RESERVED` |

## Запуск

```bash
./gradlew test
./gradlew run
```

Тесты используют `runTest`: виртуальное время делает проверку таймаутов детерминированной, а не зависящей от скорости машины.

## Почему нельзя ходить в сеть внутри транзакции

Соединение с базой занято всё время сетевого вызова: транзакция держит snapshot, мешает `VACUUM` убирать мёртвые версии строк и удерживает взятые блокировки. Внешний сервис, отвечающий 3 секунды, при пуле в 10 соединений превращается в отказ всего сервиса на третьем десятке параллельных запросов. Увидеть это можно заданием 3 недели 8: `pg_stat_activity` покажет `idle in transaction`.

## Задания

1. **Классифицированный retry.** Повторять только transient-ошибки (таймаут соединения, `503`, `429` с `Retry-After`), с exponential backoff, jitter и потолком попыток. Не повторять `4xx`-ошибки бизнес-уровня. Тест: неретраибельная ошибка вызывает gateway ровно один раз.
2. **Timeout после успеха.** Смоделировать худший случай: gateway выполнил charge, а ответ не дошёл. Проверить, что повтор с тем же ключом возвращает исходный результат и не создаёт второй платёж. Это эксперимент 7 из [docs/concurrency-track.md](../docs/concurrency-track.md).
3. **Reconciliation.** Написать джоб, который находит `RESERVED` старше N минут, спрашивает у gateway статус по idempotency key и переводит операцию в `COMPLETED` или `FAILED`. Без такого джоба состояние `RESERVED` бессмысленно.
4. **Перенести состояние в PostgreSQL.** Заменить `MemoryRepository` на таблицу с `UNIQUE(idempotency_key)` и явными переходами состояний (`RESERVED -> COMPLETED|FAILED`, обратных переходов нет). Показать, что после этого `Mutex` перестаёт быть единственной защитой и корректность сохраняется при двух instance.
5. **Outbox.** Записывать намерение отправить событие в ту же транзакцию, что и изменение данных, а отправку выполнять отдельным воркером с `SKIP LOCKED` (лаборатория очереди - `Weak7/locks-session-*.sql`, секции 4-5). Объяснить, какую проблему это решает и почему at-least-once требует идемпотентного потребителя.
6. **Блокирующий JDBC и dispatcher.** Когда repository станет настоящим, вынести блокирующие вызовы на подходящий dispatcher и объяснить, почему `Dispatchers.IO` не спасает, если пул соединений меньше числа корутин (сравните с решением в `Weak15/Application.kt`).

## Что разобрать с ментором

- Разбор использования потоков и поведения при отмене: кто именно отменяется и что происходит с уже начатой работой.
- Saga/outbox - строго на уровне выбранного сценария, без ухода в distributed transactions.
- Какие ошибки в вашем домене transient, а какие постоянные, и кто это решает.

## Критерий готовности

- Нет бесконечных retry и зависших соединений с базой.
- Повтор запроса не создаёт второй платёж - подтверждено тестом, а не рассуждением.
- Неизвестный результат внешнего вызова не превращается в `FAILED` автоматически.
