# Неделя 1 — подробная теория

Kotlin/JVM на сервере, Gradle, HTTP и жизненный цикл запроса.

> Цель недели — не написать три endpoint (это полчаса), а понять, что происходит вокруг них: как получается артефакт, кто создаёт объекты, какой поток исполняет ваш код и что случится, если этот поток заблокировать.

---

## 1. Платформа: JDK, JVM, bytecode, JAR, classpath

### 1.1 Что исполняется

Kotlin-компилятор (`kotlinc`) превращает `.kt` в `.class` с **JVM bytecode** — тем же, что генерирует `javac`. Из этого следует практическое:

- Kotlin-код и Java-код взаимозаменяемы в рантайме; Spring не знает, что вы пишете на Kotlin.
- Kotlin-специфика (nullability, data class, corutines) реализуется через аннотации, синтетические методы и сгенерированные классы; она видна рефлексией не всегда так, как в исходнике.
- **Data class без `jackson-module-kotlin`** десериализуется плохо: у него нет no-arg конструктора, а имена параметров конструктора недоступны без метаданных. Модуль решает обе проблемы, поэтому он есть в зависимостях.

**JVM** — процесс, который загружает классы, JIT-компилирует горячие методы в машинный код и управляет памятью (GC). Важные для сервера следствия:

- Первые запросы после старта медленнее: JIT ещё не прогрет. Это искажает бенчмарки — прогревайте перед измерением.
- Память процесса ≠ heap. `-Xmx` ограничивает только heap; metaspace, стеки потоков (~1 МБ на поток), direct buffers и код JIT — сверху. В контейнере с лимитом это первая причина OOMKill.

### 1.2 JAR и classpath

`JAR` — zip с `.class`, ресурсами и `META-INF/MANIFEST.MF`. Spring Boot собирает **executable JAR**: внутри лежат `BOOT-INF/classes` (ваш код) и `BOOT-INF/lib/*.jar` (зависимости), а точка входа — собственный `JarLauncher`, который умеет читать вложенные JAR.

**Classpath** — упорядоченный список источников классов. Ключевое правило: **при дублировании класса выигрывает первый найденный**. Отсюда:

- Конфликт версий не ловится компилятором. Собралось с версией 2.0, в рантайме подгрузилась 1.4 → `NoSuchMethodError`/`NoClassDefFoundError`.
- Диагностика: `./gradlew dependencies` и `./gradlew dependencyInsight --dependency <name>` показывают, откуда пришла версия и кто её «поднял».
- Spring Boot BOM (`io.spring.dependency-management` или platform) фиксирует согласованный набор версий — поэтому в `build.gradle.kts` у зависимостей Spring нет версий.

### 1.3 Gradle

Три фазы:

1. **Initialization** — читается `settings.gradle.kts`, определяется состав проектов.
2. **Configuration** — выполняется весь `build.gradle.kts` каждого проекта. Здесь строится граф задач. Тяжёлый код здесь замедляет *любую* команду, включая `./gradlew help`.
3. **Execution** — выполняются только запрошенные задачи и их зависимости.

Задачи для этой недели:

| Команда | Что делает |
|---|---|
| `./gradlew compileKotlin` | компиляция main |
| `./gradlew test` | компиляция test + запуск JUnit |
| `./gradlew bootRun` | запуск приложения из classpath проекта |
| `./gradlew bootJar` | собирает executable JAR в `build/libs` |
| `./gradlew build` | `assemble` + `check` |

**Toolchain** (`kotlin { jvmToolchain(17) }`) говорит Gradle: компилировать и запускать тестами именно на JDK 17, скачав его при необходимости. Это отвязывает сборку от того, какая JDK стоит у разработчика, — воспроизводимость.

**Wrapper** (`gradlew`, `gradle/wrapper/gradle-wrapper.properties`) фиксирует версию самого Gradle. Правило: в репозиторий коммитятся и `gradlew`, и `gradle-wrapper.jar`; собирать всегда через `./gradlew`, не через локальный `gradle`.

## 2. Серверный процесс против мобильного приложения

Это главный сдвиг мышления недели, и он не про синтаксис.

### 2.1 Время жизни

Мобильное приложение живёт от запуска до свёртывания; ОС может убить процесс, и это нормальный сценарий. Backend-процесс живёт неделями. Следствия:

- **Утечка памяти не прощается.** Кеш без ограничения размера, статический `MutableList`, ThreadLocal без очистки — на клиенте почти незаметны, на сервере приводят к OOM через несколько дней.
- **Ленивая инициализация опасна иначе:** первый запрос после деплоя может быть на порядок медленнее.

### 2.2 Конкурентность

На клиенте есть один пользователь и один UI-поток. На сервере одновременно исполняются сотни запросов, и все они видят одни и те же бины.

```kotlin
@RestController
class Bad {
    private var lastName: String? = null      // ОБЩЕЕ состояние!

    @GetMapping("/hello")
    fun hello(@RequestParam name: String): String {
        lastName = name                        // гонка между запросами
        return "Hello, $lastName"
    }
}
```

Два параллельных запроса могут получить чужое имя. Правило: **бин по умолчанию singleton; его поля должны быть неизменяемыми зависимостями, а не данными запроса**. Данные запроса живут в параметрах метода и в локальных переменных (они на стеке потока, а значит изолированы).

### 2.3 Блокировка потока

На клиенте блокирующий вызов в main thread — подтормаживание анимации. На сервере он занимает поток пула, который в это время не обслуживает никого другого. Пул на 200 потоков и запрос на 2 секунды дают потолок ~100 rps; при пуле в 4 потока — 2 rps, и это ровно то, что показывает задание с `/slow`.

## 3. HTTP как контракт

### 3.1 Stateless

HTTP не хранит состояние между запросами: сервер не обязан помнить, что было в предыдущем. Всё, что нужно для обработки, приходит в самом запросе — URL, заголовки, тело, cookie, токен.

Зачем это нужно: любой инстанс может обработать любой запрос → горизонтальное масштабирование и перезапуск без потери сессий. Как только вы кладёте состояние в память процесса (`HttpSession` в heap, локальный кеш авторизации), вы теряете это свойство и получаете «липкие» сессии.

### 3.2 Методы и коды

| Метод | Safe | Idempotent | Типичный успех |
|---|---|---|---|
| GET | да | да | 200 |
| HEAD | да | да | 200 |
| PUT | нет | да | 200 / 204 |
| DELETE | нет | да | 204 |
| POST | нет | **нет** | 201 (+ `Location`) / 200 |
| PATCH | нет | обычно нет | 200 |

Коды, которые пригодятся сразу:

- `400` — синтаксически/семантически невалидный запрос; `422` — валидный формат, но неприемлемое содержимое (используйте одно из двух последовательно).
- `401` — не аутентифицирован (кто ты?); `403` — аутентифицирован, но нет прав (не твоё).
- `404` — нет ресурса; иногда возвращают вместо `403`, чтобы не раскрывать существование чужого объекта.
- `409` — конфликт состояния (например, версия устарела).
- `429` — превышен лимит; `503` + `Retry-After` — сервис временно недоступен.

### 3.3 Идемпотентность против повторяемости

- **Повторяемость** — можно ли технически отправить тот же запрос ещё раз. Отправить можно любой.
- **Идемпотентность** — гарантия контракта: n повторов дают тот же итог, что и один.

`POST /echo` повторяем, но не идемпотентен по контракту: сервер вправе создать второй ресурс. Это и есть корень всей финтех-части курса: клиент, не дождавшийся ответа из-за таймаута, не знает, выполнилась ли операция. Решение (неделя 7) — `Idempotency-Key` + UNIQUE constraint, то есть перенос гарантии в базу.

### 3.4 Кеширование

`Cache-Control: max-age=30` на `/hello` — это заявление: ответ можно переиспользовать 30 секунд без обращения к серверу. Клиент, CDN и прокси имеют право так поступить. Отсюда практическое правило: **персональные ответы кешировать без `private`/`no-store` нельзя** — иначе прокси отдаст ответ одного пользователя другому.

Валидационные заголовки: `ETag` + `If-None-Match` → `304 Not Modified` без тела. Тот же `ETag`/`version` позже станет механизмом оптимистической блокировки (недели 2 и 7).

## 4. Spring Boot: что создаётся и когда

### 4.1 Аннотация

```kotlin
@SpringBootApplication
class Application

fun main(args: Array<String>) {
    runApplication<Application>(*args)
}
```

`@SpringBootApplication` разворачивается в три:

- `@Configuration` — сам класс является источником определений бинов;
- `@ComponentScan` — сканируются пакеты **начиная с пакета этого класса** (частая ошибка: контроллер лежит выше по дереву пакетов и не находится);
- `@EnableAutoConfiguration` — подключаются автоконфигурации из `META-INF/spring/*.imports` зависимостей.

### 4.2 Автоконфигурация

Каждый auto-configuration класс обвешан условиями:

```
@ConditionalOnClass(DispatcherServlet::class)      // есть ли класс на classpath
@ConditionalOnMissingBean(ObjectMapper::class)      // не создал ли пользователь свой
@ConditionalOnProperty("app.feature.enabled")       // включено ли свойством
```

Ментальная модель: **зависимость на classpath = включённая функция; ваш собственный бин = отключённый дефолт**. Проверить, что именно сработало: `--debug` при запуске печатает Condition Evaluation Report (positive/negative matches).

### 4.3 Dependency injection

```kotlin
@RestController
class HttpController(private val clock: Clock)   // конструкторная инъекция
```

Почему только конструктор:

1. Зависимости `val` → бин неизменяем и потокобезопасен.
2. Обязательные зависимости видны в сигнатуре, объект нельзя собрать «наполовину».
3. Класс тестируется без Spring: `HttpController(Clock.systemUTC())`.
4. Нет циклических зависимостей «по недосмотру» — цикл падает на старте, а не работает случайно.

Field injection (`@Autowired lateinit var`) ломает все четыре пункта и в Kotlin вдобавок требует `lateinit`.

### 4.4 Конфигурация и профили

Порядок источников (позднее переопределяет раннее): `application.yaml` → `application-<profile>.yaml` → переменные окружения → аргументы командной строки. Профиль включается `--spring.profiles.active=local` или `SPRING_PROFILES_ACTIVE=local`.

Правило для будущих недель: **секреты — только через переменные окружения**, никогда в `application.yaml` в репозитории.

## 5. Путь запроса: от сокета до контроллера

```
1. TCP-соединение принимает Tomcat acceptor
2. NIO poller ждёт готовности данных
3. Запрос отдаётся worker-потоку из пула (server.tomcat.threads.max, по умолчанию 200)
4. Filter chain (в т.ч. Spring Security — неделя 9)
5. DispatcherServlet — фронт-контроллер
6. HandlerMapping находит метод по URL/методу/заголовкам
7. HandlerAdapter резолвит аргументы:
     @RequestParam / @PathVariable / @RequestBody (через HttpMessageConverter)
8. Вызов вашего метода контроллера
9. Возврат: объект → HttpMessageConverter (Jackson) → JSON → response buffer
10. Исключение? → HandlerExceptionResolver / @ControllerAdvice (неделя 2)
11. Поток возвращается в пул
```

**Проверка себя:** поставьте breakpoint внутри метода контроллера и прочитайте stack trace целиком снизу вверх — там видны все перечисленные шаги, включая имя потока (`http-nio-8080-exec-3`).

### 5.1 Модель «поток на запрос»

Пока метод контроллера исполняется, поток занят. Он не переключится на другой запрос, даже если ждёт сеть или базу. Это делает модель простой (ThreadLocal, MDC, transaction context работают) и одновременно уязвимой: медленный внешний вызов исчерпывает пул.

Экспериментальная проверка (задание недели):

```properties
server.tomcat.threads.max=4
```

```bash
for i in $(seq 1 20); do curl -s localhost:8080/slow & done
time curl -s -o /dev/null -w '%{http_code} %{time_total}\n' localhost:8080/health
```

`/health` не виноват в медленном `/slow`, но страдает от него — потому что ресурс общий. Это первое знакомство с темой «saturation» (неделя 12) и мотивация для корутин и таймаутов (неделя 11).

### 5.2 Очередь и таймауты

Когда все worker-потоки заняты, соединения ждут в очереди ОС (`accept-count`). Клиент этого не видит — он видит рост latency, потом таймаут. Правило: **лимит должен быть явным и коротким** (таймауты на HTTP-клиенте, на запросах к БД, на транзакциях), иначе очередь растёт до отказа.

## 6. Тест как контракт

```kotlin
@WebMvcTest
class HttpControllerTest(@Autowired val mvc: MockMvc) {
    @Test fun health() {
        mvc.get("/health").andExpect { status { isOk() } }
    }
}
```

`MockMvc` поднимает Spring MVC без реального сокета и Tomcat: быстро, но проверяет ровно то, что относится к контракту — маршрут, статус, заголовки, форму тела. Что он **не** проверяет: настройки connector'а, реальную сериализацию на уровне сети, фильтры, не зарегистрированные в тестовом контексте.

Формулируйте тест как утверждение о контракте, а не о реализации: «`GET /hello?name=kotlin` возвращает 200 и `Cache-Control: max-age=30`», а не «метод вызывает сервис».

## 7. Типичные ошибки недели

1. Контроллер лежит вне пакета `@SpringBootApplication` → 404 на все маршруты.
2. Изменяемое поле в контроллере/сервисе → гонка, воспроизводится только под нагрузкой.
3. `data class` без `jackson-module-kotlin` → падение десериализации или `null` в non-null поле.
4. Возврат `200` на создание вместо `201` + `Location`.
5. Проверка «работает» только через IDE; `./gradlew bootJar && java -jar` не запускается из-за отсутствующего профиля или ресурса.
6. Логирование тела запроса целиком — на неделе 9 это станет утечкой персональных данных.

## 8. Критерий готовности

- Рисуете request lifecycle от TCP до контроллера и обратно, без подсказок.
- Собираете и запускаете приложение без IDE.
- Объясняете, какой поток исполняет ваш код и что произойдёт, если его заблокировать.
- Отличаете идемпотентность от повторяемости на примере `POST /echo`.
