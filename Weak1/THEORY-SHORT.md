# Неделя 1 — краткая теория

**Тема:** Kotlin/JVM на сервере, Gradle, HTTP и жизненный цикл запроса.
**Результат:** поднять Spring Boot-приложение и объяснить путь запроса от сокета до контроллера и обратно.

---

## 1. JVM и сборка

- **JDK** = JVM (среда исполнения) + компилятор и инструменты. Kotlin компилируется в тот же **bytecode**, что и Java.
- **JAR** — zip-архив с классами и ресурсами. **Fat/boot JAR** Spring Boot дополнительно содержит зависимости и свой загрузчик классов.
- **Classpath** — упорядоченный список мест, где JVM ищет классы. Две версии одной библиотеки на classpath → побеждает первая, отсюда `NoSuchMethodError` в рантайме, а не при компиляции.
- **Gradle lifecycle**: `initialization` → `configuration` (выполняется весь `build.gradle.kts`) → `execution` (выполняются только нужные таски). Граф задач: `compileKotlin` → `classes` → `test` → `bootJar`.
- **Toolchain** (`jvmToolchain(17)`) фиксирует версию JDK для сборки независимо от JDK разработчика. **Wrapper** (`./gradlew`) фиксирует версию Gradle.

## 2. Сервер против мобильного приложения

| | Клиент (iOS/KMP) | Backend-процесс |
|---|---|---|
| Время жизни | сессия пользователя | недели, до деплоя |
| Параллелизм | один пользователь, main thread | сотни одновременных запросов |
| Состояние | принадлежит пользователю | общее и конкурентное |
| Утечка памяти | одно устройство | копится до падения процесса |
| Блокировка потока | лагает анимация | занят поток пула, страдают чужие запросы |

Следствие: **изменяемое поле в singleton-бине — разделяемое состояние**, а **любая блокирующая операция — занятый поток пула**.

## 3. HTTP

- **Stateless**: сервер не обязан помнить предыдущий запрос; всё состояние передаётся явно (токен, cookie, тело).
- Коды: `2xx` успех, `3xx` перенаправление/кеш (`304`), `4xx` ошибка клиента (`400`, `401`, `403`, `404`, `409`, `422`, `429`), `5xx` ошибка сервера.
- **Идемпотентность** — повторное выполнение не меняет результат по сравнению с однократным. `GET`, `PUT`, `DELETE` идемпотентны; `POST` — нет. Идемпотентность ≠ безопасность (safety) и ≠ повторяемость запроса.
- Заголовки контракта: `Content-Type`, `Accept`, `Cache-Control`, `ETag`, `Location`, `Idempotency-Key`, `X-Request-Id`.

## 4. Spring Boot

- `@SpringBootApplication` = `@Configuration` + `@ComponentScan` + `@EnableAutoConfiguration`.
- **Автоконфигурация** — набор `@Conditional`-классов: бин создаётся, если есть класс на classpath и нет ручного бина.
- **DI через конструктор** — единственный правильный вариант: неизменяемые зависимости, обязательность видна в сигнатуре, класс тестируется без Spring.
- Порядок старта: `SpringApplication.run` → создание `ApplicationContext` → component scan → auto-configuration → создание бинов → запуск встроенного Tomcat.

## 5. Потоки и путь запроса

```
socket → Tomcat connector (acceptor/poller)
       → worker thread из пула (по умолчанию 200)
       → Filter chain
       → DispatcherServlet
       → HandlerMapping → HandlerAdapter
       → @RestController метод
       → HttpMessageConverter (Jackson) → response
```

Один запрос = один поток пула на всё время обработки (классический blocking-стек). Пул исчерпан → новые запросы ждут в очереди `acceptCount` → таймауты у клиента.

## Шпаргалка

- Блокирующий вызов не «медленный», он **дорогой**: он забирает общий ресурс.
- Contract живёт не только в JSON body: статус-код и заголовки — часть API.
- Не хранить состояние запроса в полях бина; передавать параметрами.
- Собирать проект без IDE — обязательный навык: `./gradlew bootJar && java -jar build/libs/*.jar`.

## Контрольные вопросы

1. **Почему HTTP называется stateless?** Каждый запрос самодостаточен; сервер не обязан хранить контекст между запросами, что позволяет масштабировать его горизонтально.
2. **Чем идемпотентность отличается от повторяемости?** Повторяемость — что запрос технически можно отправить снова. Идемпотентность — что n повторов дают тот же итоговый эффект, что и один.
3. **Что такое classpath и почему возможен конфликт зависимостей?** Список источников классов; при дублировании классов используется первый найденный, поэтому транзитивные версии библиотек могут молча подменять друг друга.
4. **Что произойдёт при `server.tomcat.threads.max=4` и 20 медленных запросах?** Четыре обрабатываются, остальные стоят в очереди; быстрый `/health` перестаёт отвечать мгновенно.
