# Неделя 4-1 — подробная теория (Spring API поверх индексов)

B-tree с точки зрения приложения: как контракт API, выбор ключей и способ пагинации превращаются в стоимость страниц на диске.

Физическое устройство индекса подробно разобрано в [Weak4/THEORY-DETAILED.md](../Weak4/THEORY-DETAILED.md); здесь — минимум и приложение.

---

## 1. Минимум про B-tree

- Индекс — файл из страниц по 8 КБ: **root** → **internal** → **leaf**.
- В листьях лежат отсортированные значения ключа и **TID** (`ctid`) версий строк. Листья связаны двусвязным списком.
- Поиск: спуск по дереву (3-4 чтения страниц на реальных объёмах) → чтение страницы heap по TID → проверка видимости версии в snapshot.
- Диапазон: спуск + последовательный проход по листьям. Отсюда бесплатный `ORDER BY` и ранняя остановка на `LIMIT`.
- Вставка в заполненную страницу → **page split**. Монотонный ключ → split 90/10 и плотный индекс, случайный → split 50/50 и распухание.
- Индекс не знает о видимости строк, поэтому обычный Index Scan обращается к heap.

## 2. Выбор первичного ключа — решение уровня API

### 2.1 Три варианта

| | `bigint identity` | `uuid` v4 | `uuid` v7 / ULID |
|---|---|---|---|
| Размер ключа | 8 байт | 16 байт | 16 байт |
| Кто генерирует | БД | клиент/приложение | клиент/приложение |
| Точка вставки в индекс | правый край | случайная | почти правый край |
| Размер индекса на 1 млн строк | наименьший | заметно больше | близок к монотонному |
| `correlation` heap↔ключ | ≈ 1 | ≈ 0 | ≈ 1 |
| Раскрывает объём бизнеса | да | нет | частично (время) |
| Идемпотентность на клиенте | нет | да | да |

Строка «кто генерирует» важнее, чем кажется. Идентификатор, сгенерированный клиентом, позволяет сделать создание ресурса идемпотентным: повтор `POST` с тем же id не создаст второй объект (неделя 7). За это платят структурой индекса — если только не взять v7.

### 2.2 Как измерить разницу

Задание: две таблицы, одинаковые данные, разные PK. Загрузить по миллиону строк из приложения `batchUpdate`, затем:

```sql
SELECT relname, pg_size_pretty(pg_relation_size(indexrelid)) AS idx_size
FROM pg_stat_user_indexes WHERE relname IN ('bench_bigint', 'bench_uuid');

SELECT tablename, attname, correlation FROM pg_stats
WHERE tablename IN ('bench_bigint','bench_uuid') AND attname = 'id';
```

Ожидаемое: индекс на `uuid` v4 существенно больше, время загрузки заметно дольше, корреляция около нуля.

### 2.3 Внешний идентификатор

Частая и хорошая схема: внутренний `bigint` PK для FK и индексов + внешний `uuid`/ULID в API. Внутри — компактно и плотно, наружу — не раскрывается объём. Цена — один дополнительный уникальный индекс.

## 3. Пагинация как решение о плане запроса

### 3.1 Offset

```sql
SELECT * FROM payments WHERE account_id = :acc
ORDER BY created_at DESC LIMIT 20 OFFSET 100000;
```

База обязана получить 100 020 строк в нужном порядке и выбросить первые 100 000. Стоимость растёт линейно с глубиной. Плюс — при конкурентных вставках строки «переезжают» между страницами: пользователь видит дубли или пропуски.

### 3.2 Keyset

```sql
SELECT id, created_at, amount_minor
FROM payments
WHERE account_id = :acc
  AND (created_at, id) < (:lastCreatedAt, :lastId)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

С индексом `(account_id, created_at DESC, id DESC)`:

- фильтр `account_id` — равенство по ведущей колонке;
- `(created_at, id) < (...)` — диапазон, начинающийся с точной точки в листе;
- порядок обеспечен индексом → сортировки нет;
- `LIMIT 20` останавливает выполнение после 20 строк.

Стоимость не зависит от глубины. Кортежное сравнение `(a, b) < (x, y)` — именно то, что ложится на составной индекс; разложение в `a < x OR (a = x AND b < y)` планировщик обрабатывает хуже.

Курсор для клиента — base64 от `(created_at, id)`, непрозрачная строка. Не отдавайте наружу сырые внутренние значения без необходимости.

### 3.3 Сортировка и LIMIT

Практическое правило: **`ORDER BY` без поддерживающего индекса означает сортировку всей выборки**. При превышении `work_mem` — external merge sort на диск, что видно в `EXPLAIN ANALYZE` как `Sort Method: external merge  Disk: 24MB`. Это одна из самых частых причин «неожиданно медленного списка».

### 3.4 Фильтры «по любому полю»

UI с десятью фильтрами провоцирует создать десять индексов. Правильный ответ:

1. Собрать реальную статистику использования фильтров.
2. Индексировать те комбинации, которые действительно встречаются, составными индексами (неделя 5).
3. Оставшиеся редкие комбинации пусть идут Seq Scan/Bitmap — это нормально.

Ограничьте контракт: не обещайте в API произвольные комбинации фильтров и сортировок, если не готовы их поддержать индексами.

## 4. Цена индекса, видимая в endpoint'е

### 4.1 Запись

Каждый индекс — дополнительная запись при `INSERT`, дополнительный WAL и возможный page split. В endpoint'е `POST /api/payments` это прямой вклад в p99 latency.

Эксперимент из приложения: прогнать нагрузочный сценарий создания платежей с 0, 1 и 4 индексами, сравнить throughput и latency. Это тот же write-overhead тест, что в SQL-лаборатории, но в терминах, на которые смотрит продукт.

### 4.2 HOT-update

Если `UPDATE` не трогает ни одну индексируемую колонку и новая версия влезает в ту же страницу, индексы не обновляются (Heap-Only Tuple). Отсюда правило проектирования: **часто обновляемые поля (`status`, `updated_at`, счётчики) индексировать дорого вдвойне**. Если индекс на `status` всё-таки нужен, рассмотрите partial-форму (неделя 5).

### 4.3 Неиспользуемые индексы

```sql
SELECT relname, indexrelname, idx_scan,
       pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;
```

Индекс с нулём сканирований на живой системе платит за запись, место, кеш и VACUUM, не давая ничего. Удаление — `DROP INDEX CONCURRENTLY`. Осторожно: уникальные индексы, поддерживающие constraint, удалять нельзя, даже если `idx_scan = 0`.

### 4.4 Создание индекса на живой системе

`CREATE INDEX` берёт блокировку, мешающую записи. На боевой таблице всегда `CREATE INDEX CONCURRENTLY`: дольше, два прохода, не блокирует запись, но не работает внутри транзакции и может оставить `INVALID` индекс при сбое (нужно проверить `pg_index.indisvalid` и пересоздать). Это же правило переедет в миграции на неделях 8 и 13.

## 5. Как приложение должно измерять

### 5.1 Реалистичные данные

`EXPLAIN ANALYZE` на 10 строках всегда покажет Seq Scan — и это правильно. Все выводы делаются на объёме, сравнимом с боевым, и с распределением, похожим на боевое (skew по `status`, «тяжёлые» аккаунты).

### 5.2 Прогрев

Первое выполнение читает страницы с диска, второе — из `shared_buffers`. Сравнивайте однородные запуски и смотрите на `BUFFERS`:

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
-- Buffers: shared hit=812 read=17
```

`read` — реально прочитанные страницы, `hit` — попадания в кеш.

### 5.3 Медленные запросы

- `log_min_duration_statement` на стороне PostgreSQL — самый простой способ увидеть кандидатов.
- `pg_stat_statements` — агрегированная статистика по нормализованным запросам (`calls`, `total_exec_time`, `mean_exec_time`, `rows`). Подключается расширением; станет основным инструментом на неделе 12.
- На стороне приложения — фиксировать длительность запроса вместе с `requestId`, чтобы связать HTTP и SQL.

### 5.4 Массовая загрузка

`batchUpdate` вместо цикла из отдельных `INSERT`; ещё быстрее — `COPY` через `CopyManager` JDBC-драйвера. Индексы на большой таблице выгоднее создавать после загрузки данных.

## 6. Лаборатория недели со стороны приложения

1. Сгенерировать 1 000 000 платежей через `batchUpdate` с реалистичным распределением `account_id`, `status`, `created_at`.
2. Реализовать `GET /api/accounts/{id}/payments` в двух вариантах — offset и keyset — и сравнить планы и latency на глубине 100 000.
3. Создать индексы на `uuid`, `bigint`, `timestamptz`, `status` и сравнить `pg_relation_size(indexrelid)`.
4. Найти долю выборки, при которой планировщик переключается на Seq Scan; проверить влияние `random_page_cost`.
5. Замерить latency `POST /api/payments` при 0, 1 и 4 индексах.
6. Найти неиспользуемые индексы через `pg_stat_user_indexes` и обосновать удаление каждого.

## 7. Типичные ошибки недели

1. Индекс «на всякий случай» под каждый фильтр из UI.
2. Offset-пагинация в контракте ленты, обнаруженная как проблема только в проде.
3. Выводы о планах на тестовой базе в тысячу строк.
4. `ORDER BY` по полю, которого нет в индексе, вместе с `LIMIT` — и удивление от disk sort.
5. `CREATE INDEX` без `CONCURRENTLY` на живой таблице.
6. `uuid` v4 PK, выбранный по привычке с клиента, без замера последствий.
7. Отсутствие индекса на FK-колонке — обнаруживается блокировками на неделе 7.

## 8. Критерий готовности

- Для каждого индекса в проекте называете конкретный запрос, который он ускоряет, и операции, которые он замедляет.
- Объясняете разницу offset и keyset планами, а не словами.
- Понимаете, что Index Scan дополнительно читает heap, и знаете условия, при которых этого можно избежать.
- Показываете эксперимент, где планировщик сознательно не использует индекс, и объясняете, почему это верное решение.
