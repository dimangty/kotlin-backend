# Неделя 4-1 — краткая теория (Spring API поверх индексов)

Та же неделя, что и [Weak4](../Weak4/), но со стороны приложения: как выбор ключей и индексов в API отражается на B-tree.

---

## 1. Минимум про B-tree

- Дерево страниц по 8 КБ: root → internal → **leaf**; в листьях — отсортированные ключи и **TID** версий строк.
- Листья связаны в список → дешёвые диапазоны, `ORDER BY` и `LIMIT` без сортировки.
- Высота на миллионе строк — 3 уровня. Поиск = 3 чтения индекса + чтение heap + проверка видимости.
- Вставка в полную страницу → **page split**; монотонный ключ даёт split 90/10, случайный — 50/50.

## 2. Решения API, которые бьют по индексу

| Решение в API | Что происходит в B-tree |
|---|---|
| PK — `uuid` v4 | случайные вставки, распухший индекс, `correlation ≈ 0` |
| PK — `bigserial` | плотный индекс, но id предсказуемы |
| PK — `uuid` v7 / ULID | компромисс: генерируется клиентом и почти монотонен |
| Offset-пагинация | `OFFSET 100000` читает и выбрасывает 100 000 строк |
| Keyset-пагинация | ложится на индекс, стоимость не растёт с глубиной |
| Сортировка по полю без индекса | сортировка всей выборки, возможен disk sort |
| Фильтр по «любому» полю из UI | соблазн добавить индекс на каждое поле |

## 3. Keyset-пагинация

```sql
SELECT id, created_at, amount_minor
FROM payments
WHERE account_id = :acc AND (created_at, id) < (:ts, :id)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

Индекс `(account_id, created_at DESC, id DESC)` обслуживает и фильтр, и порядок, и `LIMIT`. Это правильный контракт для ленты и ledger.

## 4. Цена индекса со стороны приложения

- Каждый индекс замедляет запись; в endpoint'е создания платежа это прямой рост latency.
- `UPDATE` индексируемой колонки лишает строку HOT-оптимизации.
- Неиспользуемый индекс видно в `pg_stat_user_indexes.idx_scan = 0` — его надо удалять.

## 5. Как измерять из приложения

- Логировать медленные запросы (`spring.jpa.properties.hibernate.session.events.log.LOG_QUERIES_SLOWER_THAN_MS` / собственный interceptor JDBC).
- Прогонять `EXPLAIN (ANALYZE, BUFFERS)` на **боевом объёме**, а не на тестовой таблице в 10 строк.
- Массовую загрузку делать `batchUpdate`/`COPY`, индексы строить после данных.

## Контрольные вопросы

1. **Почему `uuid` v4 в качестве PK дорог?** Вставки идут в случайные страницы индекса: split 50/50, распухание, случайный I/O, потеря корреляции с физическим порядком.
2. **Чем keyset-пагинация лучше offset?** Она превращается в диапазонный поиск по индексу с `LIMIT`, поэтому стоимость постоянна и результат стабилен при конкурентных вставках.
3. **Почему `LIMIT 20` иногда бесплатен, а иногда нет?** Если порядок обеспечен индексом, выполнение останавливается после 20 строк; если требуется сортировка, сначала обрабатывается вся выборка.
4. **Что показывает `idx_scan = 0`?** Индекс не используется ни одним запросом, но продолжает платить цену за запись, место и VACUUM.
