# Неделя 3-1 — краткая теория (Spring + JDBC поверх схемы)

Та же неделя, что и [Weak3](../Weak3/), но со стороны приложения: где проходит граница между HTTP, JDBC и ограничениями базы.

---

## 1. Что нужно помнить из SQL-части

- Таблица (**heap**) — страницы по 8 КБ; строка хранится как **версия** (tuple) с адресом `ctid` и системными полями `xmin`/`xmax`.
- **UPDATE не меняет строку на месте**: создаётся новая версия, старая помечается мёртвой → bloat → `VACUUM`.
- **MVCC**: каждая транзакция видит snapshot; читатели не блокируют писателей.
- **ANALYZE** обновляет статистику, от которой зависит план запроса.
- Деньги — `bigint` в минорных единицах; время — `timestamptz`.

## 2. Путь запроса до диска

```
HTTP → Controller → Service → JdbcTemplate/DataSource
     → соединение из HikariCP
     → PostgreSQL backend-процесс
     → парсинг → planner (использует статистику) → executor
     → shared buffers → страницы heap/индекса
```

Каждое соединение пула = отдельный серверный процесс PostgreSQL. Пул меньше, чем кажется: `maximumPoolSize` порядка (2 × ядра + диски) обычно лучше, чем 100.

## 3. Где какой инвариант

| Уровень | Что защищает | Чего не может |
|---|---|---|
| Bean Validation на DTO | форма запроса (длина, диапазон, формат) | конкурентность, состояние |
| Домен/сервис | правила, зависящие от состояния | другой инстанс приложения, ручной SQL |
| Constraint в БД | абсолютный инвариант при любой конкуренции | дать красивое сообщение пользователю |

Дублирование здесь — не избыточность, а слои с разной зоной действия.

## 4. Ошибки базы в HTTP-коды

| Исключение | SQLSTATE | Ответ |
|---|---|---|
| `DuplicateKeyException` | 23505 unique_violation | `409 Conflict` |
| `DataIntegrityViolationException` | 23503 FK, 23514 CHECK | `409` / `422` |
| `EmptyResultDataAccessException` | — | `404` |

Наружу отдаётся только `code`/`message`/`requestId` — никогда текст исключения БД (в нём имена таблиц и constraint'ов).

## 5. JDBC-минимум

- `JdbcTemplate`/`NamedParameterJdbcTemplate` — явный SQL, явный маппинг в DTO.
- **Только параметризованные запросы** (`?` / `:name`). Конкатенация строк = SQL injection.
- `INSERT ... RETURNING id` — получить сгенерированный ключ одним обращением.
- Без явной транзакции каждый оператор коммитится сам (autocommit) — на неделях 6-8 это станет критично.

## 6. Что смотреть в проекте

- `POST /api/accounts`, `POST /api/payments` — валидация DTO и constraint базы на одном сценарии.
- `GET /api/accounts/{id}/physical-tuple` — `ctid`/`xmin` до и после UPDATE.
- `pg_relation_size` / `pg_indexes_size` после генерации 100 000 платежей.

## Контрольные вопросы

1. **Почему нельзя полагаться только на проверку в сервисе?** Между `SELECT` и `INSERT` состояние меняется, а инстансов приложения несколько; уникальность гарантирует только UNIQUE-индекс.
2. **Почему `ctid` нельзя хранить в приложении?** Он меняется при каждом UPDATE и при `VACUUM FULL` — это адрес версии, а не идентификатор строки.
3. **Что произойдёт, если пул соединений больше, чем способна обслужить БД?** Каждое соединение — процесс PostgreSQL; избыток даёт борьбу за CPU и память, latency растёт, а throughput падает.
4. **Почему `SELECT *` в JDBC-коде — плохая идея?** Ломается при изменении схемы, тянет TOAST-поля и скрывает реальную стоимость запроса.
