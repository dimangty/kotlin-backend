# Неделя 12 — краткая теория

**Тема:** логи, метрики и диагностика производительности.
**Результат:** уметь ответить, какой запрос медленный, почему он медленный и кого он блокирует.

---

## 1. Structured logging

- Лог — **событие с полями**, а не строка. Формат JSON, чтобы его можно было фильтровать и агрегировать.
- Обязательные поля: `timestamp`, `level`, `logger`, `message`, `requestId`, `userId` (если есть), `operation`, `durationMs`, `outcome`.
- **MDC** (Mapped Diagnostic Context) — ThreadLocal-хранилище, поля из которого автоматически попадают в каждую запись. Обязательно очищать в `finally`, иначе значения утекут на следующий запрос из того же потока.
- При переходе в другой поток/корутину MDC не переносится (нужен `MDCContext`).

**Что не логировать:** пароли, токены, номера карт, полные тела с ПДн, содержимое `Authorization`.

## 2. Корреляция

| ID | Смысл | Живёт |
|---|---|---|
| `requestId` | один HTTP-запрос | от фильтра до ответа |
| `traceId` | цепочка вызовов между сервисами | сквозь всю систему |
| `operationId` | одна бизнес-операция (перевод) | может пережить несколько запросов и retry |

`requestId` приходит из заголовка `X-Request-Id` или генерируется; возвращается в ответе и в теле ошибки (неделя 2).

Связка в PostgreSQL: `application_name` или комментарий в SQL (`/* requestId=... */`) позволяет найти запрос в логах базы.

## 3. Метрики

Четыре золотых сигнала: **latency, traffic, errors, saturation**.

| Метрика | Тип | Зачем |
|---|---|---|
| `http.server.requests` | timer | latency по endpoint/статусу |
| `hikaricp.connections.pending` | gauge | **saturation пула** — ждут соединение |
| `hikaricp.connections.usage` | timer | сколько держим соединение |
| длительность транзакции | timer | ловит «долгие транзакции» |
| `retry`/`deadlock`/`serialization_failure` | counter | конкурентность |
| `pending_transfers_older_than_2m` | gauge | зависшие операции (неделя 11) |

**Перцентили, а не среднее.** Считайте p50/p95/p99: среднее скрывает хвост, который и видит пользователь. Не усредняйте перцентили между инстансами — используйте гистограммы (`percentiles-histogram`).

**Кардинальность:** никогда не кладите в тег `userId`, `accountId`, URL с параметрами — это взрывает хранилище метрик.

## 4. Actuator, Micrometer, Prometheus

- **Micrometer** — фасад для метрик (`Counter`, `Timer`, `Gauge`, `DistributionSummary`).
- **Actuator** — endpoint'ы `/actuator/health`, `/metrics`, `/prometheus`, `/info`. Обязательно **закрыть авторизацией** и не выставлять наружу.
- **Prometheus** — pull-модель: сам ходит за метриками; `rate()`, `histogram_quantile()` в PromQL.

## 5. Диагностика на стороне БД

```sql
-- топ по суммарному времени
SELECT calls, mean_exec_time, total_exec_time, rows, query
FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 20;
```

- `log_min_duration_statement` — медленные запросы в лог.
- `log_lock_waits = on`, `deadlock_timeout` — долгие ожидания и дедлоки.
- `pg_stat_activity` + `pg_blocking_pids()` — кто кого блокирует прямо сейчас.
- `pg_stat_user_tables` — dead tuples, последний VACUUM.

Сортировать `pg_stat_statements` по `total_exec_time`, а не по `mean_exec_time`: частый лёгкий запрос вреднее редкого тяжёлого.

## 6. Симптомы для алертов

| Симптом | Что означает |
|---|---|
| рост p99 при неизменном RPS | план ухудшился или появились ожидания |
| `connections.pending > 0` | пул исчерпан |
| рост deadlock/serialization failure | конкуренция или неудачная модель |
| долгие транзакции, `idle in transaction` | код держит транзакцию |
| рост dead tuples, отставший VACUUM | bloat |
| replication lag | реплики не успевают |

## Контрольные вопросы

1. **Почему среднее время ответа бесполезно?** Оно скрывает хвост распределения; пользователь чувствует p95/p99, а не среднее.
2. **Почему MDC надо очищать?** Поток возвращается в пул; невычищенные значения попадут в логи следующего запроса.
3. **Почему нельзя класть `userId` в тег метрики?** Каждое уникальное значение тега создаёт отдельный временной ряд — кардинальность растёт неограниченно.
4. **Как связать HTTP-запрос, транзакцию и SQL?** Единый `requestId` в MDC логов приложения, в теле ошибки и в комментарии/`application_name` для запросов к БД.
