# Неделя 4 — краткая теория (SQL-first)

**Тема:** B-tree — устройство индекса и процесс поиска.
**Результат:** индекс перестаёт быть «магической аннотацией» и становится структурой данных с понятной ценой.

---

## 1. Что такое B-tree в PostgreSQL

- Сбалансированное дерево из страниц по 8 КБ: **root** → **internal** → **leaf**.
- В internal-страницах — разделительные ключи и ссылки на дочерние страницы.
- В **leaf**-страницах — отсортированные ключи и **TID** (`ctid`) соответствующих версий строк в heap.
- Leaf-страницы связаны в двусвязный список → диапазонный поиск и `ORDER BY` без сортировки.
- Высота на миллионах строк — обычно 3-4 уровня. Поиск = столько же обращений к страницам + обращение к heap.

## 2. Как идёт поиск

```
root page → выбрать интервал по ключу → internal page → ... → leaf page
→ найти ключ бинарным поиском внутри страницы
→ получить TID → прочитать страницу heap → проверить видимость (MVCC)
```

Ключевое: **Index Scan почти всегда читает ещё и heap** — чтобы получить остальные колонки и проверить видимость версии.

## 3. Page split и порядок ключей

- Вставка в заполненную leaf-страницу → **split**: страница делится, ключ поднимается наверх.
- **Монотонный ключ** (`bigserial`, `created_at`) → вставки в правый край, split 90/10, плотный индекс, хорошая корреляция с физическим порядком.
- **Случайный ключ** (`uuid` v4) → вставки по всему индексу, split 50/50, распухание, случайный I/O, плохая корреляция.

## 4. Цена чтения

- Страница ищется сначала в `shared_buffers`, потом в page cache ОС, потом на диске.
- **Sequential access** дешевле random: `seq_page_cost = 1.0` против `random_page_cost = 4.0` (для SSD часто снижают до 1.1).
- Отсюда: при выборке большой доли таблицы Seq Scan **дешевле** индекса, и планировщик прав, когда его выбирает.

## 5. Selectivity, cardinality, correlation

- **Cardinality** — число различных значений (`n_distinct`).
- **Selectivity** — доля строк, проходящих предикат. Индекс полезен примерно до 5-10% выборки.
- **Correlation** — насколько физический порядок строк совпадает с порядком ключа (`pg_stats.correlation`, от -1 до 1). Высокая корреляция резко удешевляет Index Scan.

Поэтому индекс на `boolean`/`status` сам по себе редко полезен: селективность 1/2 или 1/3.

## 6. Цена индекса

Каждый индекс:

- замедляет `INSERT`/`UPDATE`/`DELETE` (запись в индекс + WAL);
- ломает HOT-update, если индексируется изменяемая колонка;
- занимает место и вытесняет полезные данные из кеша;
- требует обслуживания (bloat, `REINDEX`).

```sql
SELECT pg_size_pretty(pg_relation_size('payments'))  AS heap,
       pg_size_pretty(pg_indexes_size('payments'))   AS idx;
```

## Контрольные вопросы

1. **Что хранится в leaf page B-tree?** Отсортированные значения ключа и TID (`ctid`) версий строк в heap, плюс ссылки на соседние leaf-страницы.
2. **Почему «логарифмическая сложность» не описывает реальную стоимость?** Стоимость определяется числом прочитанных страниц и тем, случайные это чтения или последовательные, а также обращениями к heap и проверкой видимости — а не числом сравнений.
3. **Почему индекс ухудшает INSERT и UPDATE?** Каждая запись должна быть отражена во всех индексах, это дополнительные страницы, WAL и возможные page splits; UPDATE индексируемой колонки лишает строку HOT-оптимизации.
4. **Почему индекс на низкоселективной колонке обычно бесполезен?** Он вернёт слишком много TID; чтение heap по ним случайными обращениями дороже, чем последовательное сканирование таблицы.
