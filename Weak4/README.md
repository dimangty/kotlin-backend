# Неделя 4. B-tree изнутри

**Результат недели:** индекс перестаёт быть аннотацией и становится структурой данных с понятной ценой. `lab.sql` создаёт миллион строк и сравнивает поиск по bigint, UUID, timestamp и низкоселективному status. `write-overhead.sql` показывает, сколько за это платят INSERT и UPDATE.

## Запуск

```bash
docker compose up -d
docker compose exec postgres psql -U study -d indexes
```

`lab.sql` выполняется автоматически при первом старте. Полный сброс - `docker compose down -v`.

Если порт 5432 на машине уже занят другой базой, задайте свободный:

```bash
PG_PORT=55432 docker compose up -d
```

## Часть 1. Чтение (`lab.sql`)

Выполняется при старте; чтобы посмотреть вывод - `docker compose logs postgres`.

Что нужно увидеть своими глазами:

- поиск существующего UUID до индекса - Seq Scan, после - Index Scan;
- `status = 'DONE'` остаётся Seq Scan даже при наличии индекса, потому что predicate попадает примерно в треть таблицы;
- `pg_relation_size` против `pg_indexes_size`.

## Часть 2. Запись (`write-overhead.sql`)

```bash
docker compose exec -T postgres psql -U study -d indexes -v ON_ERROR_STOP=1 < write-overhead.sql
```

Три одинаковые таблицы, одни и те же 300 000 строк, разное число индексов. Порядок величин на PostgreSQL 17 (абсолютные числа зависят от машины, важны пропорции):

| Индексов | INSERT | WAL | Индексы на диске |
|---|---|---|---|
| 0 | ~130 ms | ~43 MB | 0 |
| 1 (монотонный bigint) | ~240 ms | ~63 MB | ~18% от heap |
| 4 (включая случайный UUID) | ~1400 ms | ~139 MB | ~100% от heap |

Отсюда два вывода недели: индекс на случайном UUID стоит принципиально дороже индекса на монотонном ключе, а четыре индекса могут весить столько же, сколько сами данные.

Скрипт также измеряет `n_tup_hot_upd`. На таблице, залитой bulk INSERT, доля HOT-обновлений оказывается около 1,5% - и одинаковой для таблицы без индексов тоже. Это значит, что здесь HOT ломают не индексы, а отсутствие свободного места на страницах.

## Задания

1. Сохранить планы до/после индекса как артефакты недели, а не только посмотреть их в консоли.
2. Нарисовать на бумаге root/internal/leaf pages и пройти поиск конкретного ключа по своему рисунку.
3. Изменить распределение `status` (например, 1% `FAILED`) и найти точку, в которой planner переключается с Seq Scan на Index Scan.
4. Выполнить `ALTER TABLE wo_many SET (fillfactor = 70)`, перезалить данные и повторить UPDATE: объяснить, что изменилось в `hot_pct` и почему.
5. Удалить `wo_many_public_id_idx` и повторить INSERT: какая часть общей цены приходилась на один случайный UUID.

## Что разобрать с ментором

- Последствия случайного UUID как первичного ключа против монотонного bigint.
- Почему `Buffers: shared hit` важнее, чем `Execution Time`, при сравнении планов.
- Какие из четырёх индексов в `wo_many` вы бы оставили в реальном сервисе и на основании какого запроса.

## Критерий готовности

- Без заучивания объясняешь, почему индекс на boolean обычно бесполезен сам по себе.
- Понимаешь, что обычный Index Scan дополнительно читает heap.
- Для каждого созданного индекса можешь назвать и запрос-выгодоприобретатель, и цену на записи в цифрах.
