# Неделя 4 — подробная теория (SQL-first)

B-tree: устройство индекса и процесс поиска.

> Цель недели — научиться рассуждать об индексе как о структуре данных на диске: сколько страниц он занимает, сколько страниц читает поиск, что происходит при вставке и почему планировщик иногда сознательно им не пользуется.

---

## 1. Устройство B-tree

### 1.1 Дерево из страниц

Индекс — такой же файл из страниц по 8 КБ, как и таблица. Страницы бывают трёх ролей:

- **root** — одна, точка входа (её адрес хранится в метастранице);
- **internal** — навигационные: разделительные ключи и ссылки вниз;
- **leaf** — листовые: сами ключи в отсортированном порядке и **TID** (`ctid`) версий строк в heap.

PostgreSQL использует вариант B⁺-дерева (B-link tree Lehman & Yao): все данные — в листьях, листья связаны в **двусвязный список**, а sibling-ссылки позволяют читать индекс конкурентно с вставками без блокировки всего дерева.

```
                    [ root ]
              k<500 |  |  | k>=900
        +-----------+  |  +-----------+
        v              v              v
   [internal]     [internal]     [internal]
        |              |              |
        v              v              v
 [leaf: 1..99] <-> [leaf: 100..199] <-> [leaf: 200..299] <-> ...
    ключ → TID        ключ → TID          ключ → TID
```

### 1.2 Высота и ветвление

В страницу 8 КБ помещаются сотни ключей (для `bigint` — порядка 400-500 на internal-странице). Поэтому:

| Строк | Примерная высота |
|---|---|
| 1 000 | 2 |
| 1 000 000 | 3 |
| 100 000 000 | 4-5 |

Практический вывод: **разница между таблицей на миллион и на сто миллионов строк — один-два лишних чтения страницы в индексе**. Рост стоимости при масштабировании приходит не отсюда, а из обращений к heap и из объёма выборки.

### 1.3 Что лежит в листе

Пара `(значение ключа, TID)`. TID — это `(номер страницы heap, номер item)`, то есть адрес **версии** строки. Отсюда два принципиально важных факта:

1. Индекс **не знает о видимости**. В нём есть записи и на те версии строк, которые ваша транзакция не должна видеть. Поэтому после чтения индекса почти всегда нужно сходить в heap и проверить `xmin`/`xmax`.
2. При `UPDATE` версия строки меняет `ctid`, значит во **все** индексы таблицы добавляются новые записи (исключение — HOT-update, когда изменённые колонки не индексируются и новая версия влезла в ту же страницу).

Дедупликация (с PostgreSQL 13) сжимает повторяющиеся ключи в листе в одну запись со списком TID — это заметно уменьшает индексы по низкоселективным колонкам.

## 2. Процесс поиска

### 2.1 Точечный поиск

```sql
SELECT * FROM payments WHERE id = 742391;
```

1. Прочитать метастраницу → адрес root.
2. Root: бинарный поиск среди разделительных ключей → ссылка вниз.
3. Internal-страницы: то же самое, пока не дойдём до leaf.
4. Leaf: бинарный поиск ключа внутри страницы → TID.
5. Прочитать страницу heap по TID.
6. Проверить видимость версии в snapshot текущей транзакции.
7. Вернуть строку.

Итого при высоте 3: 3 чтения страниц индекса + 1 чтение heap. Плюс, если индекс не уникальный, — обход соседних записей.

### 2.2 Диапазонный поиск

```sql
SELECT * FROM payments WHERE created_at >= '2026-01-01' AND created_at < '2026-02-01';
```

Спуск до первого подходящего ключа, дальше — **последовательный проход по связанным leaf-страницам**, пока ключ удовлетворяет условию. Это же свойство даёт бесплатный `ORDER BY created_at` и эффективный `LIMIT`: планировщик может остановиться, набрав нужное число строк.

### 2.3 Bitmap Scan

Когда TID много и они разбросаны, читать heap по одному невыгодно. PostgreSQL строит **bitmap** страниц, которые нужно посетить, сортирует его и читает heap по возрастанию номера страницы — то есть превращает случайный I/O в почти последовательный. Цена: теряется порядок строк, а при нехватке `work_mem` bitmap становится «lossy» (запоминается страница, а не конкретный item) и добавляется `Recheck Cond`. Подробнее — неделя 5.

### 2.4 Index Only Scan (анонс недели 5)

Если все нужные колонки есть в индексе **и** страница heap помечена в visibility map как «все версии видимы всем», обращение к heap пропускается. Отсюда неочевидное: index-only scan зависит от того, давно ли отрабатывал `VACUUM`.

## 3. Вставка, page split и порядок ключей

### 3.1 Split

Вставка идёт в конкретную leaf-страницу по значению ключа. Если места нет:

1. страница делится на две;
2. записи распределяются между ними;
3. новый разделительный ключ поднимается в родительскую страницу;
4. если и там нет места — split идёт выше, вплоть до роста высоты дерева.

Split — дорогая операция: несколько страниц изменяются, всё пишется в WAL.

Оптимизация: при вставке **в правый край** индекса (монотонный ключ) PostgreSQL делает асимметричный split примерно 90/10, оставляя левую страницу почти полной. При вставке в середину — 50/50.

### 3.2 Монотонный против случайного ключа

| | `bigserial` / `created_at` | `uuid` v4 |
|---|---|---|
| Точка вставки | правый край | случайная страница |
| Split | 90/10 | 50/50 |
| Заполненность страниц | высокая | ~70%, индекс крупнее |
| I/O при вставке | горячая страница в кеше | случайные чтения по всему индексу |
| `correlation` с heap | близка к 1 | около 0 |
| Утечка информации | предсказуемые id | нет |

Практика недели: создать одну и ту же таблицу с `bigint`-ключом и с `uuid`-ключом, загрузить по миллиону строк и сравнить время загрузки и `pg_relation_size` индексов. Разница обычно кратная.

Компромисс — **UUID v7 / ULID**: глобально генерируемый идентификатор с временным префиксом, то есть почти монотонный. Он даёт свойства случайного ключа на уровне API и свойства последовательного — на уровне индекса.

### 3.3 fillfactor

`CREATE INDEX ... WITH (fillfactor = 90)` — сколько процента страницы заполнять при построении, оставляя место под будущие вставки. Для append-only данных выгодно 100, для часто обновляемых — меньше. Для таблиц `fillfactor` влияет на вероятность HOT-update.

### 3.4 Bloat индекса

Индекс тоже распухает: удалённые записи освобождаются не сразу, страницы могут стать полупустыми. Лечится `REINDEX CONCURRENTLY` (без длительной блокировки). Смотреть:

```sql
SELECT indexrelname, idx_scan, pg_size_pretty(pg_relation_size(indexrelid))
FROM pg_stat_user_indexes
WHERE relname = 'payments'
ORDER BY pg_relation_size(indexrelid) DESC;
```

`idx_scan = 0` на живой системе — кандидат на удаление: индекс платит цену записи и не приносит пользы.

## 4. Стоимость: диск, кеш, случайный и последовательный доступ

### 4.1 Иерархия

```
shared_buffers (внутри PostgreSQL)
   ↓ промах
page cache ОС
   ↓ промах
диск (SSD / HDD)
```

`EXPLAIN (ANALYZE, BUFFERS)` показывает `shared hit` (нашли в shared_buffers) и `shared read` (пришлось читать снаружи). Именно `read` — реальная цена; `hit` почти бесплатен.

### 4.2 Модель стоимости планировщика

Планировщик считает абстрактную «стоимость» из параметров:

| Параметр | Значение по умолчанию | Смысл |
|---|---|---|
| `seq_page_cost` | 1.0 | последовательное чтение страницы |
| `random_page_cost` | 4.0 | случайное чтение страницы |
| `cpu_tuple_cost` | 0.01 | обработка строки |
| `cpu_index_tuple_cost` | 0.005 | обработка записи индекса |
| `cpu_operator_cost` | 0.0025 | вычисление оператора |
| `effective_cache_size` | 4GB | сколько данных, по мнению планировщика, в кеше (ОС + PG) |

`random_page_cost = 4.0` — наследие вращающихся дисков. На SSD его обычно снижают до 1.1-2.0, и это может радикально изменить выбор плана в пользу индексов. `effective_cache_size` не выделяет память, а только влияет на оценки — заниженное значение отпугивает планировщик от индексов.

### 4.3 Почему Seq Scan бывает правильным выбором

Пусть таблица занимает 100 000 страниц и мы выбираем 30% строк. Index Scan потребует порядка сотен тысяч случайных обращений к heap — по 4 условных единицы каждое. Seq Scan прочитает 100 000 страниц последовательно по 1 единице. Планировщик считает и выбирает второе — и он прав.

Граница обычно проходит в районе 5-10% выборки, но зависит от корреляции: при `correlation ≈ 1` нужные строки лежат рядом, и Index Scan остаётся выгодным даже на больших долях.

## 5. Selectivity, cardinality, correlation

### 5.1 Определения

- **Cardinality** — число различных значений колонки. В `pg_stats.n_distinct`: положительное число — абсолютное количество; отрицательное — доля от числа строк (`-1` = все значения уникальны).
- **Selectivity** — оценённая доля строк, проходящих предикат. Считается по гистограмме, most common values (`most_common_vals` / `most_common_freqs`) и `null_frac`.
- **Correlation** — от -1 до 1, насколько физический порядок строк на диске совпадает с порядком значений колонки. Для `created_at` в append-only таблице ≈ 1; для случайного `uuid` ≈ 0.

```sql
SELECT attname, n_distinct, null_frac, correlation, most_common_vals
FROM pg_stats
WHERE tablename = 'payments';
```

### 5.2 Как это влияет на решение

```sql
-- n_distinct = 3, selectivity ≈ 0.33 → индекс сам по себе бесполезен
CREATE INDEX ON payments (status);

-- selectivity ≈ 1/1000 → индекс полезен
CREATE INDEX ON payments (account_id);
```

Индекс на низкоселективной колонке становится полезным в двух формах (неделя 5): в составе **составного** индекса (`(status, created_at)` для конкретного статуса) и как **partial** индекс (`WHERE status = 'NEW'`, когда таких строк 1%).

### 5.3 Skew

Реальные данные неравномерны: 90% платежей в статусе `DONE`, 1% — `FAILED`. Один и тот же запрос с разным значением параметра требует разных планов. PostgreSQL хранит most common values, чтобы это учитывать, но при использовании prepared statements может выбрать **generic plan** по усреднённой оценке. Диагностика — сравнить `EXPLAIN` с литералом и с параметром; управление — `plan_cache_mode`.

## 6. Цена индекса

### 6.1 Запись

Каждый `INSERT` добавляет запись в каждый индекс. Каждый `UPDATE` индексируемой колонки — тоже (и лишает строку HOT-оптимизации). Каждое из этих изменений пишется в WAL.

Эксперимент недели (обязательный):

```sql
-- 0 индексов
\timing on
INSERT INTO bench SELECT ... FROM generate_series(1, 1000000);

-- 1 индекс
CREATE INDEX ON bench (account_id);
-- повторить загрузку

-- 4 индекса
CREATE INDEX ON bench (created_at);
CREATE INDEX ON bench (status);
CREATE INDEX ON bench (external_id);
-- повторить загрузку
```

Типичный результат — замедление в 2-4 раза. Побочный вывод: при массовой загрузке индексы выгоднее создавать **после** данных.

### 6.2 Место и кеш

`pg_indexes_size` нередко превышает `pg_relation_size`. Индексы конкурируют с данными за `shared_buffers`: лишний индекс не только занимает диск, но и вытесняет из кеша полезные страницы.

### 6.3 Обслуживание

Bloat, `REINDEX`, увеличение времени `VACUUM` (он должен пройти по всем индексам). Индекс — это обязательство, а не разовое улучшение.

### 6.4 Правило

**Индекс добавляется под конкретный запрос, с планом до и после, и с оценкой стоимости записи.** «На всякий случай» — запрещено правилами плана.

## 7. Лаборатория недели

1. Сгенерировать таблицу на 1 000 000 строк с реалистичным распределением `user_id`, `status`, `created_at`.
2. Замерить точечный поиск без индекса (Seq Scan) и с индексом.
3. Создать индексы на: `uuid`, монотонный `bigint`, `timestamptz`, низкоселективный `status`. Сравнить размеры:

   ```sql
   SELECT indexrelname, pg_size_pretty(pg_relation_size(indexrelid))
   FROM pg_stat_user_indexes WHERE relname = 'bench';
   ```

4. Менять долю выборки (`WHERE amount_minor > X`) и найти точку, где планировщик переключается с Index Scan на Seq Scan. Проверить, как её сдвигает `SET random_page_cost = 1.1`.
5. Убедиться, что `SET enable_seqscan = off` — инструмент **исследования**, а не решение: сравнить оценённую и фактическую стоимость.
6. Измерить write overhead: 0 / 1 / несколько индексов.
7. Нарисовать индекс на бумаге и вручную пройти поиск конкретного ключа — это то, что проверяет ментор.

## 8. Типичные ошибки недели

1. Вывод «индекс не работает» на таблице в 1000 строк.
2. Индекс на `boolean` или `status` сам по себе.
3. Индексирование часто обновляемой колонки без учёта потери HOT-update.
4. Использование `enable_seqscan = off` в приложении.
5. Замер `EXPLAIN ANALYZE` один раз, без прогрева кеша — первое выполнение читает с диска и не сравнимо со вторым.
6. Забыть `ANALYZE` после массовой загрузки и обвинить планировщик.
7. UUID v4 в качестве PK на большой таблице без понимания последствий.

## 9. Критерий готовности

- Без заучивания объясняете, почему индекс на `boolean` обычно бесполезен сам по себе.
- Понимаете, что Index Scan дополнительно читает heap, и знаете, когда этого можно избежать.
- Показываете эксперимент, где planner сознательно не использует существующий индекс, и объясняете, почему он прав.
- Называете цену конкретного индекса в терминах записи, места и обслуживания.
