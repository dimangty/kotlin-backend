# Неделя 7 — краткая теория

**Тема:** блокировки, дедлоки и финтех-корректность.
**Результат:** управлять конкурентным изменением данных и диагностировать зависшие операции.

---

## 1. Уровни блокировок

**Table-level** (берутся автоматически DDL и DML):

| Режим | Кто берёт | Конфликтует с |
|---|---|---|
| `ACCESS SHARE` | `SELECT` | только `ACCESS EXCLUSIVE` |
| `ROW SHARE` | `SELECT FOR UPDATE/SHARE` | `EXCLUSIVE`, `ACCESS EXCLUSIVE` |
| `ROW EXCLUSIVE` | `INSERT/UPDATE/DELETE` | `SHARE` и выше |
| `SHARE` | `CREATE INDEX` (без CONCURRENTLY) | запись |
| `ACCESS EXCLUSIVE` | `ALTER TABLE`, `DROP`, `VACUUM FULL` | **всё**, включая `SELECT` |

**Row-level** — на конкретных строках, через `SELECT ... FOR ...` и при `UPDATE`/`DELETE`.

## 2. Режимы row-level блокировки

| Конструкция | Смысл |
|---|---|
| `FOR UPDATE` | эксклюзивно: я буду менять строку |
| `FOR NO KEY UPDATE` | буду менять неключевые колонки; слабее, меньше конфликтов (это и берёт обычный `UPDATE`) |
| `FOR SHARE` | строка не должна измениться, пока я работаю |
| `FOR KEY SHARE` | ключ не должен измениться (это берёт FK-проверка) |
| `NOWAIT` | не ждать — сразу ошибка `55P03` |
| `SKIP LOCKED` | пропустить занятые строки — основа очереди задач |

## 3. Optimistic против pessimistic

| | Optimistic (`version`) | Pessimistic (`FOR UPDATE`) |
|---|---|---|
| Когда лучше | конфликты редки | конфликты часты, операция дорогая |
| Стоимость конфликта | холостая работа + повтор | ожидание |
| Риск | много повторов | lock wait, deadlock |
| HTTP | `409` естественно | клиент просто ждёт |

## 4. Deadlock

**Wait-for cycle**: A держит `acc1` и ждёт `acc2`, B держит `acc2` и ждёт `acc1`.

- PostgreSQL обнаруживает цикл через `deadlock_timeout` (по умолчанию 1 с) и **откатывает одну транзакцию** с `40P01`.
- Профилактика: **единый порядок захвата ресурсов** (например, всегда по возрастанию `account_id`), короткие транзакции, минимум блокируемых строк.
- Deadlock отличается от **lock wait**: lock wait — просто ожидание, оно рассасывается; deadlock не рассосётся никогда, поэтому его разрывает СУБД.

Таймауты: `lock_timeout` (ждать блокировку), `statement_timeout` (выполнять запрос), `idle_in_transaction_session_timeout` (простой в транзакции).

## 5. Идемпотентность

```sql
CREATE TABLE idempotency_keys (
    key           text PRIMARY KEY,
    request_hash  bytea NOT NULL,
    response      jsonb,
    created_at    timestamptz NOT NULL DEFAULT now()
);
```

Протокол: `INSERT ... ON CONFLICT DO NOTHING` → если 0 строк, ключ уже есть → вернуть сохранённый результат (или `409`, если тело запроса отличается по `request_hash`).

**Идемпотентность без UNIQUE constraint не работает**: проверка «есть ли такой ключ» перед вставкой — это TOCTOU.

## 6. Ledger

- Проводки **immutable**: только `INSERT`, никаких `UPDATE`/`DELETE`.
- Double-entry: у перевода две записи (дебет и кредит), их сумма равна нулю.
- `balance_minor` — **projection**, а не истина; истина — сумма проводок.
- Отмена — новая компенсирующая проводка, а не удаление.

Один mutable balance проще, но не даёт аудита, восстановления и проверки инварианта.

## 7. Outbox (концепция)

Событие для внешнего мира записывается в таблицу **в той же транзакции**, что и бизнес-данные. Отдельный процесс читает таблицу и отправляет с at-least-once семантикой. Это снимает проблему «записали в БД, но не отправили» или наоборот.

---

## Контрольные вопросы

1. **Как возникает deadlock при переводе A→B и B→A?** Первая транзакция блокирует A и ждёт B, вторая блокирует B и ждёт A. Цикл ожиданий не разрешится сам, PostgreSQL откатывает одну.
2. **Когда optimistic lock лучше `SELECT FOR UPDATE`?** Когда конфликты редки: нет ожиданий, нет риска deadlock, а редкий конфликт дёшево разрешается повтором или `409`.
3. **Почему idempotency key должен подкрепляться UNIQUE constraint?** Иначе между проверкой существования и вставкой проходит время, и два параллельных запроса создадут две операции.
4. **Зачем `SKIP LOCKED`?** Чтобы несколько воркеров разбирали очередь без двойного захвата задания и без ожидания друг друга.
