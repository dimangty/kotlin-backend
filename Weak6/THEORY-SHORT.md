# Неделя 6 — краткая теория (SQL-first)

**Тема:** транзакции, ACID, MVCC и уровни изоляции.
**Результат:** предсказывать, какие данные видит каждая транзакция и какие аномалии допускаются.

---

## 1. ACID

| | Что гарантирует | Чем обеспечивается в PostgreSQL |
|---|---|---|
| **A**tomicity | всё или ничего | undo через MVCC + статус транзакции в clog |
| **C**onsistency | переход из корректного состояния в корректное | constraints + ваша бизнес-логика |
| **I**solation | степень изоляции от параллельных транзакций | MVCC snapshots + уровни изоляции + блокировки |
| **D**urability | закоммиченное не теряется | WAL + fsync при коммите |

Consistency — единственная буква, за которую отвечаете вы, а не СУБД.

## 2. Границы транзакции

- Autocommit: каждый оператор — своя транзакция.
- `BEGIN ... COMMIT/ROLLBACK` — явные границы.
- Правило: транзакция должна быть **короткой** и **не содержать сетевых вызовов**. Долгая транзакция держит snapshot и мешает VACUUM.
- Ошибка в транзакции переводит её в состояние `aborted`: все последующие операторы падают до `ROLLBACK`. `SAVEPOINT` позволяет откатить часть.

## 3. Snapshot

- **Read Committed** — новый snapshot на **каждый оператор**. Внутри одной транзакции два одинаковых `SELECT` могут вернуть разное.
- **Repeatable Read / Serializable** — один snapshot на всю транзакцию.
- Версия строки видима, если создавшая её транзакция закоммичена и попадает в snapshot, а погасившая — нет.

## 4. Аномалии

| Аномалия | Суть | RC | RR | S |
|---|---|---|---|---|
| Dirty read | чтение незакоммиченного | нет | нет | нет |
| Non-repeatable read | повторное чтение строки даёт другое значение | **да** | нет | нет |
| Phantom read | повторный запрос диапазона даёт другой набор строк | **да** | нет* | нет |
| Lost update | два клиента читают и перезаписывают | **да** | ошибка 40001 | ошибка 40001 |
| Write skew | разные строки, нарушен общий инвариант | **да** | **да** | ошибка 40001 |

\* В PostgreSQL Repeatable Read реализован как snapshot isolation и фантомов не даёт, в отличие от требований стандарта.

Dirty read в PostgreSQL невозможен вообще: `READ UNCOMMITTED` работает как `READ COMMITTED`.

## 5. Что делает Read Committed особенным

При `UPDATE` строки, изменённой конкурентной транзакцией, Read Committed **ждёт** её завершения, а затем **перечитывает** строку и заново применяет условие `WHERE`. Отсюда:

- `UPDATE accounts SET balance = balance - 100 WHERE id = 1` конкурентно **безопасен** (атомарное чтение-изменение);
- `SELECT balance` → вычислить в приложении → `UPDATE SET balance = :new` **теряет обновления**.

## 6. Serialization failure

Repeatable Read и Serializable при конфликте бросают ошибку:

- `40001 could not serialize access ...`
- Serializable дополнительно: `could not serialize access due to read/write dependencies`.

**Retry обязателен и должен повторять всю бизнес-транзакцию с начала** — данные, прочитанные в первой попытке, недействительны. Retry ограниченный (3-5 попыток, backoff + jitter). Повторять `40001`/`40P01` можно; `23505`, `23514`, `23503` — нельзя, это логические ошибки.

## 7. Безопасное списание — четыре способа

1. **Атомарный UPDATE с условием**: `UPDATE accounts SET balance_minor = balance_minor - :amt WHERE id = :id AND balance_minor >= :amt` + проверка числа затронутых строк.
2. **Пессимистичная блокировка**: `SELECT ... FOR UPDATE`, затем расчёт и `UPDATE`.
3. **Оптимистичная блокировка**: `UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?`.
4. **Serializable + retry**.

Плюс `CHECK (balance_minor >= 0)` как последний рубеж.

---

## Контрольные вопросы

1. **Может ли Read Committed потерять обновление?** Да — если чтение и запись разнесены (read-modify-write в приложении). Атомарный `UPDATE ... SET x = x - n` не теряет.
2. **Чем Repeatable Read отличается от Serializable в PostgreSQL?** RR даёт snapshot isolation: стабильное чтение, но допускает write skew. Serializable добавляет отслеживание зависимостей чтения/записи (SSI) и откатывает транзакции, нарушающие сериализуемость.
3. **Почему retry должен повторять всю бизнес-транзакцию?** После отката весь прочитанный в ней контекст устарел; повторить только последний оператор — значит применить решение, принятое по недействительным данным.
4. **Почему уровень изоляции не заменяет модель данных?** Serializable решает конфликты откатами; при высокой конкуренции за одну строку это превращается в лавину повторов. Правильная модель (ledger вместо hot row) снимает конфликт.
