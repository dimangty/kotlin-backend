# Неделя 2 — подробная теория

Spring MVC, REST, DTO, валидация и единый контракт ошибок.

> Задача недели — построить CRUD, который останется правильным, когда под ним появится PostgreSQL. Всё, что здесь делается «для чистоты», на неделях 6-8 станет условием корректности.

---

## 1. Зачем слои

### 1.1 Три зоны ответственности

- **Транспорт (controller)** — перевод HTTP в вызов и обратно: маршрут, коды, заголовки, сериализация, валидация формы запроса.
- **Бизнес-логика (service)** — правила предметной области, инварианты, границы транзакций (появятся на неделе 8).
- **Инфраструктура (repository, клиенты внешних сервисов)** — как и где лежат данные.

Проверка: **вы должны уметь вызвать сервис из теста, из консольной задачи и из Ktor** (неделя 14) — без Spring MVC. Если для этого нужен `HttpServletRequest`, слои перепутаны.

### 1.2 Типичные протечки

```kotlin
// Плохо: HTTP-семантика в сервисе
fun update(id: Long, dto: NoteRequest): ResponseEntity<NoteResponse>

// Плохо: бизнес-правило в контроллере
@PutMapping("/notes/{id}")
fun update(...): NoteResponse {
    if (dto.title.length > 100 && !user.isPremium) throw ...   // это правило домена
}

// Хорошо
fun update(id: Long, command: UpdateNote): Note        // сервис
```

Сервис бросает доменные исключения (`NoteNotFound`, `VersionConflict`), а перевод их в коды ответа — работа `@RestControllerAdvice`.

## 2. DTO, domain model и persistence model

### 2.1 Почему три модели

| Модель | Кто её меняет | Скорость изменений |
|---|---|---|
| DTO | требования клиентов, версии API | средняя, с ограничениями совместимости |
| Domain | требования бизнеса | по мере роста правил |
| Persistence | производительность, миграции, индексы | своя, часто без изменения контракта |

Слияние DTO и entity кажется экономией, пока не появляется первое расхождение: колонка переименована ради индекса — сломан контракт; клиенту нужно новое поле — приходится менять схему БД. Плюс два технических дефекта: ленивые ассоциации entity сериализуются в JSON (N+1 прямо в контроллере) и внутренние поля (`passwordHash`, флаги) утекают наружу.

### 2.2 Kotlin-специфика DTO

```kotlin
data class CreateNoteRequest(
    @field:NotBlank @field:Size(max = 200) val title: String,
    @field:Size(max = 10_000) val body: String = "",
)

data class NoteResponse(
    val id: Long,
    val title: String,
    val body: String,
    val version: Long,
    val createdAt: Instant,
)
```

- `@field:` обязателен: без указания target аннотация может попасть на конструкторный параметр и не увидеться валидатором.
- Non-null типы Kotlin + `jackson-module-kotlin`: отсутствующее поле → `MissingKotlinParameterException` → `400`, а не молчаливый `null`. Это хорошая, но не единственная линия защиты.
- Значения по умолчанию в DTO — удобно для эволюции, но помните: они делают поле опциональным навсегда.

### 2.3 Маппинг

Маппинг пишется руками (extension-функции) или через MapStruct. Ручные функции предпочтительнее на этом этапе: они явные, их видно в диффе PR и они не прячут потерю поля.

```kotlin
fun Note.toResponse() = NoteResponse(id, title, body, version, createdAt)
fun CreateNoteRequest.toCommand() = CreateNote(title.trim(), body)
```

## 3. Валидация

### 3.1 Bean Validation

Аннотация `@Valid` (или `@Validated`) на аргументе контроллера включает проверку до входа в метод:

```kotlin
@PostMapping("/notes")
fun create(@Valid @RequestBody request: CreateNoteRequest): ResponseEntity<NoteResponse>
```

Нарушение → `MethodArgumentNotValidException` с `BindingResult`, откуда достаются `field` и `message`. Для `@RequestParam`/`@PathVariable` нужен `@Validated` на классе, и исключение будет другим — `ConstraintViolationException`. Обработать нужно оба.

### 3.2 Три уровня защиты инварианта

1. **DTO-валидация** — форма запроса: длина, формат, диапазон. Дёшево, быстро, до бизнес-логики.
2. **Домен** — правила, зависящие от состояния: «нельзя изменить архивную заметку», «сумма перевода не больше баланса».
3. **База данных** — `NOT NULL`, `UNIQUE`, `CHECK`, `FOREIGN KEY`. Единственный уровень, который работает при конкурентном доступе и при обходе приложения.

Часто задаваемый вопрос: «не дублирование ли это?» Нет — у уровней разная зона действия. Валидация DTO не спасает от гонки; проверка в домене не спасает от второго инстанса приложения; constraint в БД не даёт хорошего сообщения пользователю. Тема раскроется на неделях 3, 7 и 10.

### 3.3 Кастомные правила

Для правил на несколько полей — валидатор класса (`@ConstraintValidator`) или явная проверка в домене. Не тащите в аннотации доступ к репозиториям: валидация, которая ходит в базу, создаёт TOCTOU-дыру (проверили — состояние изменилось — записали).

## 4. Единый контракт ошибок

### 4.1 Формат

```json
{
  "code": "NOTE_NOT_FOUND",
  "message": "Note 42 does not exist",
  "details": [],
  "requestId": "8f0c2c9a-..."
}
```

- **`code`** — стабильный машиночитаемый идентификатор. Клиент ветвится по нему, а не по тексту. Менять `code` — ломающее изменение.
- **`message`** — для человека и логов; язык и формулировка могут меняться.
- **`details`** — массив по полям для валидации.
- **`requestId`** — тот самый `X-Request-Id` из недели 1; на неделе 12 он свяжет ответ, лог и транзакцию.

Альтернатива-стандарт — RFC 9457 (`application/problem+json`) с полями `type`, `title`, `status`, `detail`, `instance`. Любой из вариантов приемлем; недопустимо — разный формат в разных endpoint'ах.

### 4.2 Реализация

```kotlin
@RestControllerAdvice
class ApiExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException::class)
    fun onValidation(e: MethodArgumentNotValidException, req: HttpServletRequest): ResponseEntity<ApiError> =
        ResponseEntity.badRequest().body(
            ApiError(
                code = "VALIDATION_FAILED",
                message = "Request is invalid",
                details = e.bindingResult.fieldErrors.map { FieldError(it.field, it.defaultMessage) },
                requestId = req.getHeader("X-Request-Id"),
            )
        )

    @ExceptionHandler(NoteNotFound::class)
    fun onNotFound(e: NoteNotFound, req: HttpServletRequest) = ...  // 404

    @ExceptionHandler(VersionConflict::class)
    fun onConflict(e: VersionConflict, req: HttpServletRequest) = ...  // 409
}
```

Важные правила:

- **Никогда не отдавать наружу** stack trace, текст SQL-исключения, имена таблиц. Это разведданные для атакующего (неделя 9, OWASP «excessive data exposure»).
- Обработчик-catch-all для `Exception` возвращает `500` с `code = "INTERNAL"` и **логирует** исключение целиком — наружу уходит только `requestId`.
- Не глотайте исключение молча: ответ `200` с пустым телом — худший из вариантов.

### 4.3 Соответствие исключений и кодов

| Ситуация | Код |
|---|---|
| Не разобрался JSON, нарушены аннотации | 400 |
| Валидная форма, неприемлемое содержимое | 422 (если приняли эту трактовку) |
| Нет аутентификации | 401 |
| Есть аутентификация, нет прав | 403 |
| Ресурса нет (или он чужой и мы это скрываем) | 404 |
| Конфликт версий / повтор уникального ключа | 409 |
| Ошибка на стороне сервера | 500 |

## 5. Дизайн REST-контракта

### 5.1 Ресурсы и методы

- URL — существительные во множественном числе: `/notes`, `/notes/{id}`.
- Действия, не укладывающиеся в CRUD, оформляются как подресурс или как явная операция: `POST /transfers`, `POST /notes/{id}/archive`. Это лучше, чем `POST /doAction?type=...`.
- `POST /notes` → `201 Created` + `Location: /notes/42` + тело.
- `DELETE /notes/{id}` → `204 No Content`. Повторный DELETE — тоже `204` (идемпотентность) либо `404`; выберите одно и задокументируйте.

### 5.2 PUT против PATCH

**PUT** — полная замена: тело содержит целевое состояние ресурса. Отсутствующее поле означает «его нет», то есть сброс. Поэтому PUT идемпотентен.

**PATCH** — частичное изменение. Проблема: в JSON нельзя отличить «поле не прислали» от «поле прислали как `null`». В Kotlin наивный DTO с nullable-полями склеивает эти случаи:

```kotlin
data class PatchNote(val title: String?, val body: String?)  // null = не менять? или очистить?
```

Варианты решения:

1. Обёртка с тремя состояниями (`Undefined | Null | Value`) — `JsonNullable` из `openapi-jackson-nullable` или собственный `sealed class`.
2. JSON Merge Patch / JSON Patch (RFC 7386 / 6902) — стандартизированная семантика.
3. Явные операции вместо PATCH — часто самый честный вариант для финтеха.

### 5.3 Pagination

**Offset**: `GET /notes?page=2&size=20`.
Плюсы: тривиально, можно прыгать на страницу. Минусы: при вставке/удалении элементы «переезжают» между страницами; `OFFSET 100000` заставляет БД прочитать и выбросить 100 000 строк (увидим на неделе 5).

**Cursor/keyset**: `GET /notes?cursor=<created_at,id>&limit=20`, запрос вида `WHERE (created_at, id) < (:ts, :id) ORDER BY created_at DESC, id DESC LIMIT 20`.
Плюсы: стабильно при конкурентных вставках, цена не растёт с глубиной, ложится на индекс. Минус: нет произвольного перехода на страницу N. Для лент и ledger (неделя 16) — правильный выбор.

Формат ответа:

```json
{ "items": [ ... ], "nextCursor": "eyJ0cyI6...", "hasMore": true }
```

Голый массив в ответе — ошибка проектирования: в него нельзя добавить метаданные без ломающего изменения.

### 5.4 Обратная совместимость

Мобильный клиент обновляется не у всех и не сразу — старая версия API живёт годами.

**Можно:** добавить необязательное поле в ответ; добавить необязательное поле в запрос; добавить новый endpoint; добавить новое значение enum, **если** клиент обязан игнорировать неизвестные (это должно быть в контракте изначально).

**Нельзя:** удалить или переименовать поле; изменить тип (`Long` → `String`); изменить смысл значения; сделать необязательное поле обязательным; изменить код ответа для существующего сценария.

Процедура для ломающего изменения: expand-contract — добавить новое поле рядом со старым, дать клиентам мигрировать, потом удалить старое. Тот же приём применяется к миграциям БД на неделе 13.

## 6. Optimistic version

```kotlin
data class Note(val id: Long, val title: String, val body: String, val version: Long)
```

Протокол:

1. Клиент читает заметку и получает `version = 7`.
2. Отправляет `PUT /notes/1` с `version = 7` (в теле или как `If-Match: "7"`).
3. Сервер сравнивает с текущей версией. Совпало → применяет изменение и делает `version = 8`. Не совпало → `409 Conflict`.

Зачем это на неделе без базы: чтобы структура была готова к неделе 6-7, где ровно этот механизм превращается в защиту от **lost update** — в SQL он выглядит как `UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?` и проверка числа затронутых строк. В JPA то же самое даёт `@Version` (неделя 8).

Даже in-memory реализация обязана обновлять версию **атомарно** (например, через `ConcurrentHashMap.compute`), иначе тест конкурентности на неделе 10 покажет потерю обновления уже здесь.

## 7. Dependency injection без field injection

```kotlin
@Service
class NoteService(private val repository: NoteRepository, private val clock: Clock)
```

Почему не `@Autowired lateinit var`:

- бин становится изменяемым;
- обязательные зависимости не видны в сигнатуре, объект можно создать «пустым»;
- тест требует Spring или рефлексии;
- циклические зависимости проходят незамеченными.

`Clock` в зависимостях — не педантизм: он делает время тестируемым, что понадобится на неделях 9 (истечение токена) и 11 (таймауты).

## 8. Типичные ошибки недели

1. Entity наружу вместо DTO → утечка полей и связка схемы с контрактом.
2. Разный формат ошибок в разных обработчиках.
3. `200 OK` с телом `{"error": ...}` вместо корректного 4xx.
4. Валидация только в DTO — «если поле прошло `@Positive`, дальше всё хорошо».
5. PATCH с nullable-полями без различения «не прислали» и «очистить».
6. Отдача массива вместо объекта в списочном endpoint.
7. `@field:` забыт у аннотаций валидации в data class — правило молча не работает. Проверяется негативным тестом.

## 9. Критерий готовности

- Каждый endpoint имеет корректный код ответа **и негативный тест** на каждую ветку ошибки.
- DTO не используется как JPA entity; маппинг явный.
- Контроллер не содержит бизнес-логики, сервис не знает про HTTP.
- Все ошибки приходят в одном формате и содержат `requestId`.
- В модели есть `version`, и конфликт версий возвращает `409`.
