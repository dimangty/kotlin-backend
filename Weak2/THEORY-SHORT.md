# Неделя 2 — краткая теория

**Тема:** Spring MVC, REST, DTO и валидация.
**Результат:** корректный CRUD без базы данных, разделение транспортного, бизнес- и инфраструктурного кода.

---

## 1. Слои и границы ответственности

| Слой | Отвечает за | Не должен знать про |
|---|---|---|
| Controller | HTTP: маршруты, коды, заголовки, сериализация | бизнес-правила, SQL |
| Service | бизнес-правила, транзакции, инварианты | `HttpServletRequest`, коды ответа |
| Repository | доступ к данным | правила предметной области |

Признак нарушения: в контроллере есть `if`, выражающий правило предметной области; в сервисе — `ResponseEntity`.

## 2. Три модели, а не одна

- **Request/Response DTO** — контракт API, меняется по правилам совместимости.
- **Domain model** — правила и инварианты, не знает про JSON и таблицы.
- **Persistence model** — то, как данные лежат в БД.

Одна общая модель означает: переименование колонки ломает мобильных клиентов, а требование API просачивается в схему БД. **DTO не должен быть JPA entity.**

## 3. Валидация

- Bean Validation: `@NotBlank`, `@Size`, `@Positive`, `@Email`, `@Valid` на аргументе контроллера.
- Ошибка валидации → `MethodArgumentNotValidException` → `400`.
- Валидация DTO **не заменяет** проверку инварианта в домене и constraint в БД: это три уровня защиты с разной зоной действия.

## 4. Единый error contract

```json
{
  "code": "VALIDATION_FAILED",
  "message": "Request is invalid",
  "details": [{ "field": "title", "reason": "must not be blank" }],
  "requestId": "r-1"
}
```

- Формируется в `@RestControllerAdvice` (`@ExceptionHandler`).
- `code` — машиночитаемый и стабильный; `message` — для человека; `details` — по полям; `requestId` — для связи с логами.
- Никогда не отдавайте stack trace и текст SQL-исключения наружу.

## 5. PUT против PATCH

| | PUT | PATCH |
|---|---|---|
| Семантика | полная замена ресурса | частичное изменение |
| Идемпотентность | да | обычно нет |
| Отсутствующее поле | сброс в значение по умолчанию | «не трогать» |

В Kotlin `null` в PATCH-DTO неоднозначен («очистить» или «не менять»). Решение — обёртка вроде `JsonNullable`/`Optional`-подобного типа либо отдельные endpoint'ы для операций.

## 6. Pagination и совместимость

- Offset-пагинация (`page`/`size`) проста, но «плывёт» при вставках и дорожает на больших offset.
- Cursor/keyset-пагинация (`cursor=<last_id|created_at>`) стабильна и дёшева — к ней вернёмся на неделях 5 и 16.
- Ответ списка всегда объект (`{ "items": [...], "next": "..." }`), не голый массив: массив некуда расширять.

**Правила совместимости:** добавлять поля можно; удалять, переименовывать, менять тип и сужать enum — нельзя. Старый мобильный клиент живёт годами.

## 7. Optimistic version

Поле `version` в модели, инкремент при каждом изменении. Клиент присылает версию, которую читал; несовпадение → `409 Conflict`. Это подготовка к неделям 6-7: тот же приём поверх БД защищает от lost update.

## Контрольные вопросы

1. **Почему DTO не должен быть JPA entity?** Иначе схема БД и контракт API становятся одним объектом: любое изменение одного ломает другое, а ленивые поля entity утекают в JSON.
2. **Когда `400`, а когда `422`?** `400` — запрос не разбирается/нарушает синтаксис и базовые ограничения; `422` — синтаксис корректен, но содержимое неприемлемо по бизнес-правилу. Главное — выбрать одну трактовку и придерживаться её.
3. **Почему field injection плохо?** Скрывает обязательные зависимости, делает бин изменяемым и требует Spring для тестов.
4. **Чем PUT отличается от PATCH при отсутствующем поле?** PUT трактует отсутствие как «поля нет» (замена целиком), PATCH — как «не менять».
