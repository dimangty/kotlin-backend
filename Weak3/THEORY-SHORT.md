# Неделя 3 — краткая теория (SQL-first)

**Тема:** PostgreSQL — схема, SQL и физическое хранение данных.
**Результат:** таблица перестаёт выглядеть списком объектов и становится набором страниц и версий строк.

---

## 1. Реляционная модель и ключи

- **Отношение** — множество кортежей; порядок строк не определён, пока нет `ORDER BY`.
- **PK** — минимальный уникальный необнуляемый идентификатор строки. Natural (`email`) против surrogate (`bigint`/`uuid`): суррогатный стабилен, натуральный несёт смысл и меняется.
- **FK** — ссылка на существующую строку; задаёт направление зависимости и поведение при удалении (`RESTRICT`, `CASCADE`, `SET NULL`).
- **UNIQUE** — реализуется уникальным индексом; единственный способ гарантировать отсутствие дублей при конкуренции.
- **CHECK** — предикат на строке (`amount_minor > 0`, `status IN (...)`).
- **NOT NULL** — самое дешёвое и самое часто забываемое ограничение.

Нормализация до 3NF — умолчание; денормализация (projection `balance_minor`) — осознанное решение, требующее механизма поддержания согласованности.

## 2. Физическое хранение

- Таблица (**heap**) — набор **страниц** по 8 КБ. Внутри страницы: header, массив item pointers, сами tuples, свободное место.
- **Tuple** — физическая версия строки. `ctid = (page, item)` — её адрес; он **меняется** при UPDATE.
- Системные колонки: `xmin` (транзакция, создавшая версию), `xmax` (удалившая/заменившая).
- **TOAST** — большие значения выносятся в отдельную таблицу и сжимаются.

## 3. MVCC в одну строку

**UPDATE не изменяет строку на месте — он создаёт новую версию и помечает старую как невидимую с определённого момента.**

Отсюда:

- читатели не блокируют писателей, писатели не блокируют читателей;
- каждая транзакция видит **snapshot**: набор версий, актуальных на её момент;
- «мёртвые» версии (dead tuples) занимают место → **bloat**;
- `UPDATE` одной колонки переписывает **всю строку** и обновляет **все** индексы (кроме случая HOT-update).

## 4. WAL, checkpoint, VACUUM, ANALYZE

| Механизм | Зачем |
|---|---|
| **WAL** | сначала пишется журнал, потом страницы; даёт durability из ACID и восстановление после сбоя |
| **checkpoint** | сброс грязных страниц на диск, чтобы восстановление не читало весь WAL |
| **VACUUM** | освобождает место от dead tuples, обновляет visibility map, предотвращает transaction ID wraparound |
| **ANALYZE** | собирает статистику распределения данных — от неё зависит выбор плана |

`VACUUM` обычно **не** возвращает место ОС, а делает его переиспользуемым; `VACUUM FULL` возвращает, но берёт `ACCESS EXCLUSIVE` lock.

## 5. SQL, который нужен сразу

- `JOIN` (`INNER`, `LEFT`) — соединение по предикату; `LEFT JOIN` + `WHERE` по правой таблице молча превращается в `INNER`.
- `GROUP BY` + агрегаты (`sum`, `count`, `avg`), фильтр групп — `HAVING`, а не `WHERE`.
- `count(*)` считает строки, `count(col)` — не-NULL значения.
- **Window functions**: `sum(...) OVER (PARTITION BY account_id ORDER BY created_at)` — агрегат без схлопывания строк. Основа для running balance в ledger.
- **CTE** (`WITH`) — именованный подзапрос; в современных версиях по умолчанию inlined (`MATERIALIZED` заставляет материализовать).

## 6. Деньги

Хранить в **минорных единицах целым числом** (`amount_minor bigint`) или `numeric`. Никогда `float/double`. Валюта — отдельной колонкой.

---

## Контрольные вопросы

1. **Почему SELECT может видеть старую версию строки?** Потому что его snapshot был взят до коммита новой версии; MVCC отдаёт ему ту версию, которая была видима на момент старта запроса/транзакции.
2. **Чем logical row отличается от physical tuple?** Логическая строка — сущность с ключом; физический tuple — одна её версия по конкретному `ctid`. У одной логической строки может одновременно существовать несколько версий.
3. **Почему ANALYZE влияет на выбор плана?** Планировщик оценивает стоимость по статистике (n_distinct, гистограммы, most common values). Без свежей статистики оценка числа строк неверна → выбирается неоптимальный план.
4. **Почему UPDATE приводит к bloat?** Каждая правка порождает новую версию строки, старая остаётся мёртвой до VACUUM, занимая страницы.
