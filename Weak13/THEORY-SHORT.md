# Неделя 13 — краткая теория

**Тема:** Docker, CI/CD и безопасный деплой миграций.
**Результат:** собирать, тестировать и разворачивать приложение воспроизводимо.

---

## 1. Multi-stage Dockerfile

```dockerfile
FROM gradle:8-jdk17 AS build
WORKDIR /src
COPY . .
RUN ./gradlew --no-daemon bootJar

FROM eclipse-temurin:17-jre-alpine
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
COPY --from=build /src/build/libs/*.jar app.jar
USER app
EXPOSE 8080
ENTRYPOINT ["java", "-XX:MaxRAMPercentage=75", "-jar", "/app/app.jar"]
```

Ключевое:

- **JRE, не JDK** в runtime-слое; образ меньше и поверхность атаки уже.
- **non-root пользователь** — обязательно.
- **Слои по частоте изменений**: зависимости отдельно от кода → кеш сборки работает.
- Версии базовых образов фиксируются (не `latest`).
- `.dockerignore` исключает `build/`, `.git/`, секреты.

## 2. JVM в контейнере

- JVM с Java 10+ уважает cgroup-лимиты. Использовать `-XX:MaxRAMPercentage`, а не `-Xmx` с абсолютным числом.
- Память процесса = heap + metaspace + стеки потоков + direct buffers + код JIT. Лимит контейнера должен быть заметно больше heap, иначе OOMKill.
- Обработка `SIGTERM` нужна для graceful shutdown → приложение должно быть PID 1 или запускаться через init.

## 3. Health, readiness, graceful shutdown

| Проба | Смысл | Реакция оркестратора |
|---|---|---|
| **liveness** | процесс жив | перезапустить |
| **readiness** | готов принимать трафик | убрать из балансировки |

```yaml
server.shutdown: graceful
spring.lifecycle.timeout-per-shutdown-phase: 20s
management.endpoint.health.probes.enabled: true
```

Порядок корректного выключения: readiness → `false` → балансировщик перестаёт слать трафик → дорабатываем текущие запросы → закрываем пул → выход.

Liveness **не должен** зависеть от базы: недоступность БД не повод перезапускать процесс в цикле.

## 4. Docker Compose

`docker compose up -d` поднимает app + PostgreSQL. Нужны: `healthcheck` у базы, `depends_on: condition: service_healthy`, именованный volume для данных, переменные окружения вместо хардкода.

## 5. CI pipeline

```
lint → build → unit tests → integration tests (Testcontainers)
     → migrations check (Flyway на пустой базе)
     → security scan (зависимости, секреты)
     → docker build → push
     → deploy → smoke tests
```

Требования: воспроизводимость (Gradle wrapper, зафиксированные версии), запрет деплоя при красных тестах, тег образа = commit SHA (не `latest`).

## 6. Expand-contract миграции

Во время деплоя одновременно работают старая и новая версии → схема должна быть совместима с обеими.

| Шаг | Релиз 1 (expand) | Релиз 2 (migrate) | Релиз 3 (contract) |
|---|---|---|---|
| Колонка | добавить nullable | заполнять, читать из новой | `NOT NULL`, удалить старую |
| Переименование | добавить новую | писать в обе | удалить старую |

Опасные операции: `SET NOT NULL` и `ADD FOREIGN KEY` (сканирование под сильной блокировкой → `NOT VALID` + `VALIDATE`), `CREATE INDEX` (→ `CONCURRENTLY`), смена типа, `UPDATE` всей таблицы (→ батчами).

В миграциях обязательно `SET lock_timeout = '3s'`: `ALTER TABLE`, ждущий блокировку, блокирует всю очередь запросов к таблице.

## 7. Два инстанса стартуют одновременно

Flyway берёт lock на таблице истории — второй инстанс ждёт. Но: миграция не должна ломать инстанс, который в этот момент работает. Отсюда — только backward-compatible миграции.

## 8. Backup и rollback

- Откат **кода** прост (предыдущий образ). Откат **схемы** — нет: down-миграции для данных ненадёжны.
- Стратегия: только совместимые миграции + бэкап перед рискованной операцией.
- Бэкап без проверенного восстановления бэкапом не является. `pg_dump` для логического, PITR (базовый бэкап + WAL) для восстановления на момент времени.

## Контрольные вопросы

1. **Почему liveness не должен проверять базу?** Недоступность БД не лечится перезапуском процесса; получится цикл рестартов вместо деградации.
2. **Почему миграция обязана быть backward-compatible?** Во время выкатки старые и новые инстансы работают одновременно с одной схемой.
3. **Чем `NOT VALID` + `VALIDATE CONSTRAINT` лучше прямого `SET NOT NULL`?** Первый шаг не сканирует таблицу, второй сканирует под слабой блокировкой, не останавливая запись.
4. **Почему тег образа не должен быть `latest`?** Невозможно понять, что развёрнуто, и невозможно детерминированно откатиться.
