# Безопасный деплой: expand-contract, rollback и restore

Раздел плана «Неделя 13» требует не только собрать image, но и уметь ответить на четыре вопроса: что произойдёт при двух одновременно стартующих instances, как выкатить обязательную колонку без длинной блокировки, как откатиться и как восстановить базу. Здесь - готовые ответы и процедуры; измерения, на которые они опираются, делаются в `Weak8/migration-lab.sql`.

## Правило совместимости

Во время выкатки в системе одновременно живут **две версии приложения** и **одна схема базы**. Значит, схема обязана быть совместима и со старой, и с новой версией кода. Отсюда всё остальное:

- миграция никогда не выкатывается в том же релизе, что и код, который её требует;
- удаление колонки - это отдельный, более поздний релиз, а не часть добавления;
- любой rollback приложения не должен требовать rollback схемы.

## Expand-contract в три релиза

Задача: у `payments` появляется обязательная `currency`.

| Релиз | Схема | Код | Свойство |
|---|---|---|---|
| N (expand) | `ADD COLUMN currency char(3)` - nullable, без `NOT NULL` | старая версия работает, новая пишет оба варианта | откат кода безопасен: колонка никому не мешает |
| N+1 (backfill + migrate) | бэкофилл батчами, затем `CHECK ... NOT VALID` -> `VALIDATE` -> `SET NOT NULL` | новая версия читает и пишет `currency` | ACCESS EXCLUSIVE держится доли секунды, а не время скана |
| N+2 (contract) | удалить старую колонку/дефолт, если они были | код больше не знает о старом поле | выполняется, когда откат на релиз N уже невозможен |

Ключевая деталь второго шага измерена в `Weak8/migration-lab.sql` на таблице в 500 000 строк:

| Способ | Время | Какая блокировка |
|---|---|---|
| `ALTER COLUMN ... SET NOT NULL` напрямую | ~40 мс (полный скан) | ACCESS EXCLUSIVE на всё время скана |
| `ADD CONSTRAINT ... CHECK ... NOT VALID` | ~0.4 мс | ACCESS EXCLUSIVE на каталог |
| `VALIDATE CONSTRAINT` | ~17 мс | SHARE UPDATE EXCLUSIVE - чтение и запись работают |
| `SET NOT NULL` после валидного CHECK | ~0.4 мс | ACCESS EXCLUSIVE на каталог |

На 500 000 строк разница выглядит как 40 мс против 0.4 мс. На таблице в тысячу раз больше первый вариант превращается в десятки секунд полной недоступности: за ACCESS EXCLUSIVE встают в очередь даже `SELECT`.

Правила бэкофилла: батчи по 10-50 тысяч строк, короткие транзакции, возможность остановиться в любой момент, `VACUUM (ANALYZE)` после - каждая обновлённая строка создаёт новую версию строки.

## Обязательная страховка миграции

```sql
SET lock_timeout = '3s';
SET statement_timeout = '30s';
```

Миграция, которая не смогла взять блокировку, должна упасть и быть повторена, а не выстроить за собой очередь из запросов приложения. Без `lock_timeout` одна `idle in transaction` сессия превращает безобидный `ALTER TABLE` в остановку сервиса.

## Два instance стартуют одновременно

Flyway (недели 8, 15 и 16) берёт advisory lock на время миграции, поэтому второй instance ждёт и затем видит уже применённую версию - схема не ломается. Что действительно нужно проверить:

- readiness не должен отвечать `UP`, пока миграции не завершены: иначе балансировщик пришлёт трафик в instance, работающий на старой схеме;
- при падении instance посреди миграции остаётся запись с `success = false` в `flyway_schema_history`; следующий старт откажется продолжать, пока её не разберут руками - это правильное поведение, а не ошибка;
- миграции должны быть идемпотентны по эффекту: повторный запуск той же версии не выполняется, но повторный запуск batch-бэкофилла обязан быть безопасным.

## Rollback

Откатывается **приложение**, а не база.

1. Вернуть предыдущий image tag (для этого нужны неизменяемые теги/digest, а не `latest`).
2. Проверить readiness и ключевые метрики из недели 12.
3. Схему не трогать: релиз N спроектирован так, что старая версия кода на новой схеме работает.

Rollback схемы допустим только для миграций, не менявших данные. Обратная миграция, которая удаляет колонку, уничтожает данные, записанные новой версией, - это уже не откат, а инцидент.

Репетиция отката - часть выкатки: если процедуру ни разу не выполняли, её нет.

## Backup и restore

- логический бэкап: `pg_dump -Fc` даёт снимок, пригодный для восстановления в другую базу;
- восстановление: `pg_restore -d <db> --clean --if-exists`;
- point-in-time recovery требует базового бэкапа плюс WAL-архива - это уровень инфраструктуры, а не приложения.

Проверка, которая имеет значение: **восстановление, а не создание бэкапа**. Бэкап, из которого никогда не поднимали базу, статуса бэкапа не имеет. Учебная процедура выполняется в любой неделе с PostgreSQL - например в `Weak8` после прогона `migration-lab.sql`:

```bash
docker compose exec -T postgres pg_dump -U study -Fc data > backup.dump
docker compose exec -T postgres psql -U study -d postgres -c 'CREATE DATABASE restore_check'
docker compose exec -T postgres pg_restore -U study -d restore_check < backup.dump
docker compose exec -T postgres psql -U study -d restore_check -c '\dt'
```

Зафиксировать: сколько заняло восстановление и совпадают ли контрольные суммы (`count(*)` и `sum(amount_minor)`) с исходной базой.

## Чеклист перед выкаткой

- [ ] миграция и код, который её требует, разведены по разным релизам;
- [ ] у миграции выставлены `lock_timeout` и `statement_timeout`;
- [ ] длинных `UPDATE` на всю таблицу нет, бэкофилл идёт батчами;
- [ ] индексы на живой таблице строятся через `CREATE INDEX CONCURRENTLY`;
- [ ] readiness не отвечает `UP` до завершения миграций;
- [ ] предыдущий image tag доступен, процедура отката отрепетирована;
- [ ] restore из свежего бэкапа выполнен и проверен запросом, а не фактом наличия файла.
