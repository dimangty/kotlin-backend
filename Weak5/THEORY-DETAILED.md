# Неделя 5 — подробная теория (SQL-first)

Составные и специальные индексы, `EXPLAIN (ANALYZE, BUFFERS)`.

> Неделя 4 отвечала на вопрос «как устроен индекс». Эта отвечает на вопрос «какой индекс нужен этому запросу и как это доказать». Доказательство — всегда план выполнения, а не рассуждение.

---

## 1. Составной B-tree

### 1.1 Как хранятся ключи

Индекс `(a, b, c)` — это один B-tree, ключом которого является **кортеж**. Записи отсортированы лексикографически: сначала по `a`, при равных `a` — по `b`, при равных `a, b` — по `c`.

Ментальная модель — телефонный справочник, отсортированный по (фамилия, имя, отчество). Найти всех «Ивановых» легко. Найти всех «Петровичей» — придётся прочитать справочник целиком.

### 1.2 Leftmost prefix

Индекс сужает поиск, пока предикаты идут слева направо без пропуска:

```sql
CREATE INDEX idx ON payments (user_id, status, created_at);
```

| Предикат | Что делает индекс |
|---|---|
| `user_id = 1` | спуск по первой колонке, диапазон листьев |
| `user_id = 1 AND status = 'NEW'` | спуск по двум колонкам, узкий диапазон |
| `user_id = 1 AND status = 'NEW' AND created_at > ?` | использованы все три |
| `user_id = 1 AND created_at > ?` | спуск по `user_id`, `created_at` — **фильтр** внутри диапазона |
| `status = 'NEW'` | ведущая колонка не задана → Seq Scan (или дорогой полный проход по индексу) |

Замечание: PostgreSQL умеет читать индекс целиком (`Index Scan` без `Index Cond`, только с `Filter`), если это дешевле heap. Но это не «использование индекса» в полезном смысле — сужения нет.

Про MySQL-подобный «index skip scan»: в PostgreSQL он появился только для ограниченных случаев и не является тем, на что стоит рассчитывать при проектировании.

### 1.3 Порядок колонок: правило

```
1. Колонки с равенством (=, IN)
2. ОДНА колонка с диапазоном (>, <, BETWEEN)
3. Колонки для ORDER BY
```

После range-колонки индекс перестаёт сужать поиск: значения следующих колонок внутри диапазона не отсортированы глобально. Поэтому диапазонная колонка должна быть последней из «сужающих».

Пример недели — история платежей:

```sql
SELECT id, amount_minor, status
FROM payments
WHERE user_id = :u AND created_at >= :from AND created_at < :to
ORDER BY created_at DESC
LIMIT 50;
```

Правильный индекс: `(user_id, created_at DESC)`.

- `user_id` — равенство, сужает до одного пользователя;
- `created_at` — диапазон и одновременно порядок;
- `DESC` в определении индекса позволяет идти по листьям вперёд, хотя B-tree читается и в обратную сторону — это важно в основном для многоколоночной сортировки со смешанными направлениями (`ORDER BY a ASC, b DESC`).

Сравнение, которое требует план недели:

| Индекс | Что произойдёт |
|---|---|
| `(user_id, created_at)` | спуск к пользователю, диапазон дат, порядок готов, `LIMIT` останавливает рано |
| `(created_at, user_id)` | диапазон дат по **всем** пользователям, `user_id` — фильтр; читается на порядки больше записей |

Второй индекс не бесполезен вообще — он подойдёт для отчёта «все платежи за период». Но для истории конкретного пользователя он плох, и это видно по `Rows Removed by Filter`.

### 1.4 `IN` и несколько равенств

`IN (a, b, c)` по ведущей колонке превращается в несколько спусков по дереву — это работает. `IN` по неведущей колонке — нет.

## 2. Covering index и Index Only Scan

### 2.1 INCLUDE

```sql
CREATE INDEX idx_payments_user_created
    ON payments (user_id, created_at) INCLUDE (amount_minor, status);
```

Колонки в `INCLUDE`:

- хранятся **только в листьях**, не в internal-страницах;
- не участвуют в поиске и сортировке;
- не имеют ограничений на тип оператора;
- **не влияют на уникальность** — поэтому `UNIQUE (a) INCLUDE (b)` — легальный и полезный приём.

Зачем: сделать индекс «покрывающим» для запроса, не раздувая навигационную часть дерева.

### 2.2 Условия Index Only Scan

Index Only Scan возможен, когда выполняются **оба** условия:

1. все колонки, упомянутые в `SELECT`, `WHERE`, `ORDER BY`, есть в индексе (в ключе или в `INCLUDE`);
2. страница heap, на которую ссылается запись, отмечена в **visibility map** как «все версии на этой странице видимы всем транзакциям».

Второе условие существует потому, что **индекс не хранит информацию о видимости**. Если страница не отмечена, придётся сходить в heap — это и есть `Heap Fetches` в плане:

```
Index Only Scan using idx_payments_user_created on payments
  Index Cond: (user_id = 42)
  Heap Fetches: 0                 <- идеально
```

Visibility map обновляет `VACUUM`. Отсюда неожиданный для многих факт: **index-only scan «ломается» после интенсивной записи и «чинится» после VACUUM**. На только что загруженной таблице `Heap Fetches` будет большим, пока не выполнить `VACUUM`.

### 2.3 Когда covering оправдан

Read-heavy запрос, выполняющийся часто, с небольшим набором колонок. Цена: индекс становится шире (больше листьев, больше места и WAL при записи), и любое изменение включённой колонки требует обновления индекса.

## 3. Partial и expression индексы

### 3.1 Partial

```sql
CREATE INDEX idx_payments_pending
    ON payments (created_at)
    WHERE status = 'NEW';
```

Индекс содержит только строки, удовлетворяющие условию. Плюсы: маленький (если статус редкий — доли процента таблицы), дешёвый в записи, полностью помещается в кеш.

Условия применения:

- предикат запроса должен **логически следовать** из предиката индекса. Планировщик умеет доказывать простые импликации (`status = 'NEW'` покрывает `status = 'NEW' AND created_at > ?`), но не сложные;
- условие должно быть **иммутабельным**. `WHERE created_at > now() - interval '7 days'` — недопустимо: `now()` не иммутабельна, а индекс не пересчитывается сам;
- условие не должно меняться по смыслу вместе с бизнес-логикой (иначе после релиза индекс молча перестанет применяться).

Классический сценарий финтеха: очередь необработанных операций. Строк в статусе `NEW` мало, они «горячие», обрабатываются и уходят в `DONE`. Partial-индекс здесь на порядки меньше полного.

### 3.2 Partial unique

```sql
CREATE UNIQUE INDEX ON accounts (user_id, currency) WHERE status <> 'CLOSED';
```

Выражает инвариант «у пользователя не больше одного активного счёта в валюте», который невозможно задать обычным `UNIQUE`. Очень полезная конструкция для мягкого удаления.

### 3.3 Expression

```sql
CREATE INDEX ON users (lower(email));
CREATE INDEX ON payments ((metadata->>'external_id'));
```

Индексируется результат выражения. Работает **только** при точном совпадении выражения в запросе:

```sql
SELECT * FROM users WHERE lower(email) = lower(:email);   -- индекс используется
SELECT * FROM users WHERE email = :email;                 -- НЕ используется
```

Требование: функция должна быть `IMMUTABLE`. `now()`, `random()`, функции, зависящие от `timezone` или локали, не подходят.

Побочный эффект: PostgreSQL собирает статистику по выражению — это иногда улучшает оценки даже без использования индекса.

## 4. Виды сканирования

### 4.1 Seq Scan

Последовательное чтение всех страниц heap. Дешевле индекса при большой доле выборки. Может распараллеливаться (`Parallel Seq Scan`). Наличие Seq Scan в плане — не диагноз.

### 4.2 Index Scan

Спуск по дереву → чтение heap по каждому TID. Сохраняет порядок индекса (поэтому «бесплатен» для `ORDER BY` и хорош с `LIMIT`). Плох при большом числе строк с плохой корреляцией: каждый TID — случайное чтение.

### 4.3 Bitmap Index Scan + Bitmap Heap Scan

Два узла всегда идут парой:

1. **Bitmap Index Scan** проходит индекс и строит битовую карту нужных страниц heap.
2. **Bitmap Heap Scan** сортирует карту и читает страницы по возрастанию номера — случайный I/O превращается в почти последовательный.

Дополнительные свойства:

- **BitmapAnd / BitmapOr** позволяют скомбинировать **несколько индексов** в одном запросе. Это ответ на вопрос «нужен ли составной индекс или хватит двух одиночных»: два одиночных сработают через bitmap, но составной обычно точнее и дешевле.
- При нехватке `work_mem` карта становится **lossy** — запоминается страница целиком, и появляется строка `Recheck Cond` с реальной перепроверкой каждой строки. Признак: `Rows Removed by Index Recheck`.
- Порядок строк теряется, поэтому `LIMIT` с `ORDER BY` через bitmap потребует сортировки.

### 4.4 Index Only Scan

См. раздел 2. В плане обязательно смотреть `Heap Fetches`.

### 4.5 Как читать соединения

- **Nested Loop** — для каждой строки внешней выборки поиск во внутренней. Отлично, когда внешних строк мало и есть индекс. Катастрофа, если оценка внешних строк занижена в 1000 раз.
- **Hash Join** — строит хеш-таблицу по меньшей стороне. Хорош для больших объёмов; при нехватке `work_mem` уходит в batches на диск.
- **Merge Join** — обе стороны отсортированы (часто — по индексу).

Самая частая причина «запрос вдруг стал в 100 раз медленнее» — Nested Loop, выбранный по заниженной оценке.

## 5. Специальные типы индексов

### 5.1 GIN

Generalized Inverted Index — «много значений внутри одного поля»: `jsonb`, массивы, full-text search.

```sql
CREATE INDEX ON payments USING gin (metadata jsonb_path_ops);
SELECT * FROM payments WHERE metadata @> '{"provider":"acme"}';
```

- `jsonb_ops` (по умолчанию) поддерживает больше операторов; `jsonb_path_ops` меньше, но компактнее и быстрее для `@>`.
- Запись в GIN дороже, чем в B-tree; смягчается буфером отложенной вставки (`fastupdate`, `gin_pending_list_limit`).
- Для full-text: `to_tsvector('russian', body)` + GIN по этому выражению (или по материализованной колонке).

### 5.2 GiST

Обобщённое сбалансированное дерево для «пространственных» задач: геометрия, диапазоны (`tstzrange`), ближайшие соседи (`ORDER BY point <-> point`).

Финтех-применение — `EXCLUDE` constraint, который выражает непересекаемость периодов:

```sql
ALTER TABLE rates ADD CONSTRAINT rates_no_overlap
    EXCLUDE USING gist (currency WITH =, period WITH &&);
```

Это инвариант, который нельзя выразить ни `UNIQUE`, ни `CHECK`.

### 5.3 BRIN

Block Range INdex хранит не ключи, а сводку (min/max) по диапазону блоков — обычно 128 страниц.

```sql
CREATE INDEX ON payments USING brin (created_at) WITH (pages_per_range = 64);
```

- Размер — килобайты вместо гигабайтов.
- Работает **только при высокой корреляции**: append-only таблица по времени. Если данные перемешаны, BRIN бесполезен (в каждом диапазоне блоков будет весь спектр значений).
- Поиск даёт не точный набор строк, а набор блоков-кандидатов с последующей перепроверкой.

Идеальный кандидат — большая историческая таблица, куда пишут по времени и читают диапазонами.

### 5.4 Hash

Поддерживает только `=`. С PostgreSQL 10 корректно журналируется в WAL. На практике B-tree почти всегда не хуже, поэтому hash-индекс — редкий выбор.

## 6. EXPLAIN: как читать

### 6.1 Формы

```sql
EXPLAIN SELECT ...;                                  -- только оценка, запрос не выполняется
EXPLAIN (ANALYZE) SELECT ...;                        -- ВЫПОЛНЯЕТ запрос
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, SETTINGS) SELECT ...;
```

Осторожно: `ANALYZE` действительно выполняет запрос, включая `UPDATE`/`DELETE`. Для проверки модифицирующих запросов оборачивайте в транзакцию с `ROLLBACK`.

### 6.2 Что означают числа

```
Index Scan using idx_payments_user_created on payments
  (cost=0.43..812.11 rows=210 width=48)
  (actual time=0.028..1.902 rows=198 loops=1)
  Index Cond: ((user_id = 42) AND (created_at >= '2026-01-01'))
  Filter: (status = 'DONE')
  Rows Removed by Filter: 1043
  Buffers: shared hit=214 read=9
```

- `cost=start..total` — оценка в условных единицах. `start` — до первой строки (важно для `LIMIT`), `total` — до последней.
- `rows=` в первой скобке — **оценка**, во второй — **факт**.
- `actual time=first..last` — миллисекунды.
- `loops=N` — узел выполнялся N раз; **`actual` даны на один цикл**, умножайте на `loops`.
- `Index Cond` — что реально сузило поиск по индексу.
- `Filter` + `Rows Removed by Filter` — сколько строк прочитано зря. Большое число — сигнал, что колонка должна быть в индексе.
- `Buffers: shared hit / read` — попадания в кеш и реальные чтения. `read` — настоящая цена.

### 6.3 Главный диагностический приём

**Сравнивайте `rows` оценённые и фактические.** Расхождение на порядок и больше означает, что планировщик принимал решение вслепую. Причины:

1. устаревшая статистика → `ANALYZE`;
2. коррелирующие условия (`city = 'Москва' AND country = 'RU'`) → планировщик перемножает селективности как независимые. Лечится расширенной статистикой:

   ```sql
   CREATE STATISTICS payments_stat (dependencies, ndistinct)
       ON user_id, status FROM payments;
   ANALYZE payments;
   ```

3. skew данных и generic plan для prepared statement → `plan_cache_mode`;
4. выражения, по которым нет статистики → expression index или `CREATE STATISTICS` на выражении.

### 6.4 Что смотреть по порядку

1. Есть ли узел с расхождением оценки и факта в разы — начинать оттуда.
2. `Rows Removed by Filter` — лишняя работа.
3. `Heap Fetches` в Index Only Scan.
4. `Sort Method: external merge Disk: NkB` — сортировка не влезла в `work_mem`.
5. `Rows Removed by Index Recheck` — lossy bitmap.
6. `Buffers ... read` — реальный I/O.
7. `loops` у внутреннего узла Nested Loop.

### 6.5 Инструменты исследования

`SET enable_seqscan = off;`, `SET enable_indexscan = off;` — **только для эксперимента в сессии**, чтобы увидеть альтернативный план и его стоимость. Это никогда не решение; если план плох, чинят статистику, индекс или запрос.

## 7. Лаборатория недели

Обязательная таблица результатов — по одной строке на запрос:

| Запрос | План до | Индекс | План после | Buffers до/после | Цена записи |
|---|---|---|---|---|---|

Минимальный набор экспериментов:

1. История платежей `user_id + created_at + status`: план без индексов, затем `(user_id, created_at)`.
2. Сравнить `(user_id, created_at)` и `(created_at, user_id)` на том же запросе — показать разницу в прочитанных записях.
3. Partial index для необработанных операций; показать размер против полного индекса.
4. Добиться Index Only Scan с `Heap Fetches: 0`; затем сделать массовый UPDATE и показать, как `Heap Fetches` вырос, а `VACUUM` его починил.
5. Показать случай, где planner сознательно не использует существующий индекс (большая доля выборки), и подтвердить его правоту через `enable_seqscan = off`.
6. Показать BitmapAnd от двух одиночных индексов и сравнить с составным.
7. GIN по `jsonb` для `@>` и/или BRIN по `created_at` на большой append-only таблице.
8. Замерить write overhead: массовая вставка с 0 / 1 / несколькими индексами.

Ответ на вопрос «какой индекс» считается неполным, если в нём нет: точного SQL, распределения данных и ожидаемой селективности, предикатов и порядка, изменения плана и `BUFFERS`, цены для записи и хранения.

## 8. Типичные ошибки недели

1. Диапазонная колонка раньше колонок равенства в составном индексе.
2. Отдельные индексы на каждую колонку вместо одного составного.
3. Ожидание Index Only Scan без `VACUUM`.
4. Partial index с `now()` в предикате.
5. Expression index, не совпадающий с выражением в запросе.
6. BRIN на таблице с перемешанными данными.
7. Чтение `EXPLAIN` без `ANALYZE` и вывод «план хороший» по оценкам.
8. Игнорирование `loops` при чтении Nested Loop.
9. `enable_seqscan = off` как «исправление».

## 9. Критерий готовности

- Для каждого индекса называете запрос, который он ускоряет, и операции, которые он замедляет.
- Объясняете leftmost behavior причинно, не фразой «так принято».
- Читаете `EXPLAIN (ANALYZE, BUFFERS)` и находите расхождение оценки и факта.
- Добились `Heap Fetches: 0` и умеете объяснить, почему число может вырасти.
- Показали случай осознанного отказа планировщика от индекса.
