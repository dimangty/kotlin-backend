# Неделя 5 — краткая теория (SQL-first)

**Тема:** составные и специальные индексы, `EXPLAIN ANALYZE`.
**Результат:** индекс выбирается по конкретному запросу и подтверждается планом выполнения.

---

## 1. Составной B-tree

Индекс `(a, b, c)` хранит ключи, отсортированные сначала по `a`, внутри равных `a` — по `b`, и т. д.

**Правило ведущих колонок (leftmost):** индекс применим к предикату, если используются колонки слева направо без пропусков.

| Запрос | `(a, b, c)` |
|---|---|
| `a = ?` | да |
| `a = ? AND b = ?` | да |
| `a = ? AND b = ? AND c = ?` | да |
| `b = ?` | нет (может быть Seq Scan или дорогой index-only skip) |
| `a = ? AND c = ?` | частично: спуск по `a`, `c` — фильтр |

**Порядок колонок:** сначала колонки равенства, затем **одна** колонка диапазона, затем колонки сортировки. После range-колонки индекс перестаёт сужать поиск.

```
WHERE user_id = ? AND created_at BETWEEN ? AND ?  ORDER BY created_at DESC
→ (user_id, created_at DESC)
```

## 2. Covering index и Index Only Scan

- `CREATE INDEX ... ON t (a, b) INCLUDE (c)` — `c` хранится только в листьях, не участвует в поиске и сортировке.
- **Index Only Scan** возможен, если все нужные колонки есть в индексе **и** страница помечена в **visibility map** как «все версии видимы всем».
- Отсюда: наличие всех колонок не гарантирует index-only scan — нужен свежий `VACUUM`. В плане это видно как `Heap Fetches: N`.

## 3. Partial и expression индексы

```sql
CREATE INDEX ON payments (created_at) WHERE status = 'NEW';   -- partial
CREATE INDEX ON users (lower(email));                         -- expression
```

- Partial полезен, когда запросы всегда содержат то же условие, а строк под ним мало. Индекс меньше и дешевле в записи.
- Expression работает, только если в запросе стоит **то же выражение** (`WHERE lower(email) = ?`).
- Partial unique — способ выразить «уникально среди активных»: `CREATE UNIQUE INDEX ... WHERE status <> 'DELETED'`.

## 4. Виды scan

| Узел плана | Когда |
|---|---|
| `Seq Scan` | нет индекса или выбирается большая доля строк |
| `Index Scan` | мало строк, хорошая корреляция; читает heap по одному TID |
| `Bitmap Index Scan` + `Bitmap Heap Scan` | строк средне-много или несколько индексов объединяются; heap читается по возрастанию страниц |
| `Index Only Scan` | все колонки в индексе + свежая visibility map |

## 5. Другие типы индексов

| Тип | Для чего |
|---|---|
| **GIN** | «много значений в одном поле»: `jsonb`, массивы, full-text |
| **GiST** | геометрия, диапазоны, `EXCLUDE` constraint |
| **BRIN** | огромные таблицы с высокой корреляцией (append-only по времени); крошечный размер |
| **Hash** | только `=`, редко оправдан |

## 6. EXPLAIN

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE) SELECT ...;
```

- `cost=start..total` — оценка; `actual time=..` — факт; `rows=` оценка против `rows=` факта.
- **Расхождение оценки и факта в разы — главный сигнал**: устаревшая статистика, коррелирующие условия, skew.
- `loops=N` — значения `actual` даны **на один цикл**, умножайте.
- `Buffers: shared hit/read` — попадания в кеш и реальные чтения.
- `Rows Removed by Filter` — работа, потраченная впустую.

## Контрольные вопросы

1. **Почему `(status, user_id)` и `(user_id, status)` неэквивалентны?** Порядок задаёт сортировку ключей: индекс сужает поиск только по ведущим колонкам. Запрос по одному `user_id` не может использовать первый индекс эффективно.
2. **Чем Index Scan отличается от Bitmap Heap Scan?** Index Scan читает heap по одному TID в порядке индекса (случайный I/O, сохраняет порядок). Bitmap собирает страницы, сортирует их и читает почти последовательно, но теряет порядок и может потребовать recheck.
3. **Когда partial index безопасен и полезен?** Когда условие индекса всегда присутствует в запросах, оно неизменно, и отбираемая доля строк мала.
4. **Почему все колонки в индексе не гарантируют Index Only Scan?** Индекс не хранит информацию о видимости; без отметки в visibility map придётся идти в heap (`Heap Fetches > 0`).
