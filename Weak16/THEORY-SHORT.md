## 1. System design небольшого payment service

Порядок проектирования (он же порядок ответа на собеседовании):

1. **Требования и инварианты** — что нельзя нарушать ни при каких условиях.
2. **API contract** — endpoint'ы, коды, идемпотентность, пагинация.
3. **Схема данных** — таблицы, ключи, constraints.
4. **Границы транзакций** — что атомарно, где блокировки, в каком порядке.
5. **Индексы** — под конкретные запросы, с планами.
6. **Failure modes** — что сломается и как система это переживёт.
7. **Наблюдаемость** — какие метрики покажут проблему.
8. **Capacity** — объёмы, рост, узкие места.

## 2. Минимальный API

```
POST /auth/register | /auth/login | /auth/refresh
POST /accounts
GET  /accounts/{id}
POST /transfers            (Idempotency-Key обязателен)
GET  /transfers/{id}
GET  /accounts/{id}/ledger?cursor=...
GET  /health, /actuator/*  (закрыт авторизацией)
```

## 3. Инварианты, которые защищаются

| Инвариант | Уровень защиты |
|---|---|
| Баланс не отрицателен | `CHECK` + условный `UPDATE` |
| Сумма денег в системе постоянна | транзакция + double-entry + тест |
| Один idempotency key = одна операция | `PRIMARY KEY`/`UNIQUE` |
| Проводки неизменяемы | только `INSERT`, права на таблицу |
| Projection = сумма проводок | одна транзакция + сверка |
| Доступ только к своим счетам | ownership в запросе + тест |

## 4. Failure modes

| Сбой | Реакция системы |
|---|---|
| Клиент не дождался ответа и повторил | idempotency key → тот же результат |
| Дедлок | единый порядок блокировок; retry жертвы |
| Serialization failure | ограниченный retry всей транзакции |
| Внешний сервис недоступен | timeout, breaker, `PENDING` + reconciliation |
| Инстанс упал в середине | транзакция откатилась; `PENDING` подхватит сверка |
| База перегружена | ограничения пула, `statement_timeout`, деградация вместо отказа |
| Плохой план после роста данных | мониторинг p99, `pg_stat_statements`, `ANALYZE` |

## 5. Trade-offs, которые нужно уметь защищать

| Решение | За | Против |
|---|---|---|
| Mutable balance | просто, быстро читать | нет аудита, hot row, нечего сверять |
| Ledger | аудит, восстановление, сверка | сложнее, баланс надо считать/проецировать |
| Pessimistic lock | предсказуемо | сериализация, дедлоки |
| Optimistic lock | быстро без конфликтов | холостые повторы |
| Serializable | ловит write skew | откаты, обязательный retry |
| Outbox | атомарность с внешним миром | лишняя таблица и воркер |

## 6. Threat model и audit

- Аутентификация, ownership, negative cases (неделя 9).
- Audit trail: actor, action, object, `requestId`, время; только `INSERT`.
- В логах нет секретов и ПДн.
- Actuator и метрики закрыты.

## 7. Capacity

Считать в порядках, а не точно: сколько операций в день → RPS в пике → размер таблиц через год → размер индексов → влезает ли рабочий набор в память. Узкое место в этой архитектуре — не CPU приложения, а конкуренция за строки и I/O базы.

## Контрольные вопросы

1. **Почему ledger, а не один баланс?** Он даёт аудит, восстановление и проверяемый инвариант; баланс остаётся быстрой проекцией.
2. **Что произойдёт, если клиент повторит `POST /transfers`?** Ключ идемпотентности вернёт результат первой операции; второй перевод не создаётся.
3. **Какие ошибки система повторяет автоматически?** `40001` и `40P01` — ограниченным retry всей транзакции; нарушения constraint не повторяются.
4. **Как вы докажете, что перевод корректен?** Constraints в схеме, параллельный тест на инвариант суммы, тест идемпотентности и timeline дедлока с его устранением.
