## 1. System design небольшого payment service

### 1.1 Порядок рассуждения

Этот порядок стоит воспроизводить и в проекте, и на собеседовании:

1. **Инварианты.** Что не должно нарушаться никогда: баланс не отрицателен, деньги не появляются и не исчезают, операция не выполняется дважды.
2. **API contract.** Endpoint'ы, коды ответов, идемпотентность, пагинация, правила совместимости.
3. **Схема.** Таблицы, ключи, constraints — какой инвариант защищён каким ограничением.
4. **Границы транзакций.** Что атомарно, какие строки блокируются и в каком порядке.
5. **Индексы.** Под конкретные запросы, с планом до и после и с ценой записи.
6. **Failure modes.** Что ломается и как это переживается.
7. **Наблюдаемость.** Какие метрики покажут проблему до звонка клиента.
8. **Capacity.** Объёмы, рост, узкое место.

Ошибка новичка — начать с фреймворков и слоёв. Начинать нужно с инвариантов: всё остальное является следствием.

### 1.2 Минимальный API

```
POST /auth/register
POST /auth/login
POST /auth/refresh                       ← rotation, обнаружение повторного использования
POST /accounts
GET  /accounts/{id}                      ← ownership
POST /transfers                          ← Idempotency-Key обязателен
GET  /transfers/{id}
GET  /accounts/{id}/ledger?cursor=...    ← keyset-пагинация
GET  /health
GET  /actuator/**                        ← только для роли OPS
```

Контракт `POST /transfers`:

| Ситуация | Код | Тело |
|---|---|---|
| Создан | `201` + `Location` | результат перевода |
| Повтор с тем же ключом и телом | `200`/`201` | **тот же** результат |
| Тот же ключ, другое тело | `422` | `IDEMPOTENCY_KEY_REUSED` |
| Запрос уже обрабатывается | `409` | `IN_PROGRESS` |
| Недостаточно средств | `422` | `INSUFFICIENT_FUNDS` |
| Чужой счёт | `404` | `NOT_FOUND` |
| Нет ключа идемпотентности | `400` | `IDEMPOTENCY_KEY_REQUIRED` |

## 2. Схема и инварианты

### 2.1 Таблицы

`users`, `accounts`, `transfers`, `ledger_entries`, `idempotency_keys`, `audit_events`, `outbox` (если есть внешние интеграции).

### 2.2 Карта инвариантов

Это главный документ проекта — на нём строится ревью:

| Инвариант | Где защищён | Чем проверен |
|---|---|---|
| Баланс не отрицателен | `CHECK (balance_minor >= 0)` + `UPDATE ... WHERE balance_minor >= :amt` | тест конкурентного списания |
| Сумма денег постоянна | одна транзакция + double-entry | тест на 50 параллельных переводов |
| Один ключ = одна операция | `PRIMARY KEY (key)` на `idempotency_keys` | тест параллельных `POST` с одним ключом |
| Проводки неизменяемы | только `INSERT`; права на таблицу | тест попытки `UPDATE` |
| Сумма проводок операции = 0 | транзакция + тест | invariant-тест |
| `balance_minor` = сумма проводок | одна транзакция + сверка | тест после серии операций |
| Валюты совпадают | `CHECK`/схема + проверка в домене | негативный тест |
| Доступ только к своим счетам | ownership в SQL | негативный security-тест |
| Уникальный активный счёт в валюте | partial `UNIQUE ... WHERE status <> 'CLOSED'` | негативный тест |

Правило: **инвариант без теста и без constraint — это пожелание, а не инвариант.**

## 3. Границы транзакций и конкурентность

### 3.1 Перевод

```
TX (READ COMMITTED):
  1. claim idempotency key         INSERT ... ON CONFLICT DO NOTHING
     └ не получилось → вернуть сохранённый результат
  2. SELECT ... WHERE id IN (from, to) ORDER BY id FOR NO KEY UPDATE
  3. проверки: владение, статусы, валюты, достаточность
  4. INSERT transfers; INSERT 2 × ledger_entries
  5. UPDATE accounts (from) ... WHERE balance_minor >= amount   → проверить rowcount
     UPDATE accounts (to)
  6. INSERT audit_event
  7. UPDATE idempotency_keys SET status='DONE', response=...
COMMIT
```

Ключевые решения и их обоснования:

- **`ORDER BY id`** — единый порядок захвата, профилактика дедлока (неделя 7).
- **`FOR NO KEY UPDATE`**, а не `FOR UPDATE` — меньше конфликтов, ключ мы не меняем.
- **Условный `UPDATE`** с проверкой rowcount — атомарная проверка баланса даже на Read Committed (неделя 6).
- **`CHECK`** — последний рубеж на случай ошибки в логике.
- **Retry снаружи транзакции** на `40001`/`40P01`, с backoff и jitter.

### 3.2 Обработка очереди

```sql
SELECT id FROM payments WHERE status = 'NEW'
ORDER BY created_at LIMIT 10
FOR UPDATE SKIP LOCKED;
```

Partial-индекс `ON payments (created_at) WHERE status = 'NEW'` — под этот запрос (неделя 5).

### 3.3 Внешний вызов

TX1 (`PENDING` + ключ) → COMMIT → сеть с таймаутом → TX2 (терминальный статус + проводки). Плюс reconciliation для зависших `PENDING` (неделя 11).

## 4. Индексы

Обязательная таблица в README:

| Запрос | План до | Индекс | План после | Buffers до/после | Цена записи |
|---|---|---|---|---|---|
| ledger по счёту (keyset) | Seq Scan | `(account_id, created_at DESC, id DESC)` | Index Scan, `LIMIT` останавливает | | |

## 5. Failure modes

| Сбой | Что происходит | Как система переживает |
|---|---|---|
| Клиент повторил запрос после таймаута | два `POST` с одним ключом | idempotency key → один перевод, тот же ответ |
| Два встречных перевода | wait-for cycle | единый порядок блокировок; при дедлоке — retry жертвы |
| Serialization failure | `40001` | ограниченный retry всей транзакции |
| Внешний сервис не отвечает | timeout | breaker OPEN, статус `PENDING`, reconciliation |
| Инстанс упал в середине операции | транзакция откатилась | `PENDING` подхватит сверка; частичных списаний нет |
| Два инстанса стартуют одновременно | Flyway lock | миграции backward-compatible |
| Пул соединений исчерпан | `connections.pending > 0` | быстрый отказ по `connection-timeout`, `503` вместо зависания |
| Долгая транзакция | bloat, lock waits | `idle_in_transaction_session_timeout`, алерт |
| План деградировал после роста данных | рост p99 | `pg_stat_statements`, `ANALYZE`, разбор плана |
| Утечка токена | доступ к чужим данным | короткий access, отзыв refresh, audit |

Для каждого сценария в demo должен быть либо тест, либо описанная процедура.

## 6. Trade-offs

### 6.1 Mutable balance против ledger

| | Mutable balance | Ledger + projection |
|---|---|---|
| Чтение баланса | одна строка | projection или агрегат |
| Аудит | нет | полный |
| Восстановление после бага | невозможно | пересчёт из проводок |
| Отмена | правка числа | компенсирующая проводка |
| Конкуренция | hot row на `UPDATE` | `INSERT` не конфликтует |
| Сложность | низкая | выше |

Для финтеха ledger — не роскошь: без него нельзя ответить на вопрос «откуда взялась эта сумма».

### 6.2 Способы защиты от гонок

| Подход | Когда | Цена |
|---|---|---|
| Атомарный `UPDATE ... WHERE` | операция выражается одним оператором | нет |
| Optimistic (`version`) | конфликты редки | холостые повторы |
| Pessimistic (`FOR UPDATE`) | нужно согласованно читать несколько строк | сериализация, риск дедлока |
| Serializable + retry | инвариант охватывает незаблокированные строки (write skew) | откаты |

### 6.3 Hot row

Когда все операции бьют в одну строку (счёт мерчанта, общий счётчик), любой подход упирается в сериализацию. Ответы: ledger вместо mutable balance; шардирование счётчика на N строк с суммированием; батчинг; отложенная агрегация. Уметь назвать хотя бы два и объяснить, чем платите.

### 6.4 Outbox

Атомарность с внешним миром недостижима; outbox превращает её в at-least-once + идемпотентную обработку. Цена — таблица и воркер.

## 7. Threat model и audit

Активы: балансы, ledger, персональные данные, токены, секреты подписи.

| Угроза | Защита |
|---|---|
| Доступ к чужому счёту | ownership в SQL + негативные тесты |
| Кража токена | короткий access, rotation, обнаружение повторного использования refresh |
| Подбор пароля | Argon2id с подобранными параметрами, rate limit |
| Mass assignment | DTO с белым списком полей |
| SQL injection | только параметризованные запросы |
| Утечка через логи | маскирование, аудит логов |
| Инсайдерское изменение данных | immutable ledger, audit trail, ограниченные права |

Audit trail: actor, action, object, `requestId`, IP, время; только `INSERT`; пишется в той же транзакции, что и действие.

## 8. Capacity

Считайте в порядках:

1. Операций в день → средний RPS → пиковый RPS (обычно ×3-5 от среднего).
2. Размер строки × число строк в год → размер таблиц; плюс `pg_indexes_size`.
3. Влезает ли рабочий набор (горячие данные + индексы) в `shared_buffers` и в память машины.
4. Сколько операций в секунду упирается в одну строку (hot row) — это реальный потолок, а не CPU приложения.
5. Рост ledger: append-only, значит нужен план на партиционирование или архивацию.

Узкое место такой архитектуры почти всегда — конкуренция за строки и I/O базы, а не приложение.

## 10. Mock interview: вопросы, к которым надо быть готовым

**Индексы**

- Что лежит в leaf page B-tree и как идёт поиск?
- Почему `(user_id, created_at)` и `(created_at, user_id)` неэквивалентны?
- Когда планировщик сознательно не использует индекс — и почему он прав?
- Чем Index Scan отличается от Bitmap Heap Scan?
- Почему все колонки в индексе не гарантируют Index Only Scan?
- Сколько стоит индекс для записи?

**Транзакции**

- Может ли Read Committed потерять обновление? Покажите на timeline.
- Чем Repeatable Read отличается от Serializable в PostgreSQL?
- Что такое write skew и как его поймать?
- Почему retry должен повторять всю бизнес-транзакцию?
- Как возникает дедлок при переводе A→B и B→A и как его предотвратить?
- Когда optimistic лучше `FOR UPDATE`?

**Архитектура**

- Почему idempotency key должен подкрепляться UNIQUE constraint?
- Зачем ledger, если есть баланс?
- Почему нельзя вызывать внешний сервис внутри транзакции?
- Как вы поймёте, что запрос стал медленным, и как найдёте причину?
- Как сделать миграцию `NOT NULL` на таблице в миллиард строк?

Правило ответа: **сначала факт, потом объяснение, потом чем докажете**. «Read Committed теряет обновление при read-modify-write — вот timeline — вот тест, который это ловит».

## 11. Чек-лист готовности к рабочей задаче

- [ ] Находите endpoint, проходите по слоям до SQL, объясняете transaction boundary.
- [ ] Пишете безопасную миграцию и проверяете её на копии данных.
- [ ] Читаете `EXPLAIN (ANALYZE, BUFFERS)` и не добавляете индекс без доказательства.
- [ ] Объясняете устройство B-tree и влияние порядка колонок.
- [ ] Воспроизводите и исправляете lost update и deadlock.
- [ ] Определяете блокирующую транзакцию и долгий запрос.
- [ ] Пишете интеграционный тест с PostgreSQL и конкурентный тест.
- [ ] Понимаете, где нужны retry, идемпотентность и UNIQUE constraint.
- [ ] Собираете Docker-образ, запускаете сервис, проверяете health.
- [ ] Запрашиваете review до изменения критичного финансового сценария.
